## Introduction

The textbook's fidelity spectrum runs from abstract representations through simulated prototypes to live data products. This tutorial traverses that spectrum in a single project, using a scenario drawn from real design research.

Price et al.'s *When Looking Out of the Window Is Not Enough* studied households with rooftop solar panels. The central problem: occupants cannot intuitively connect what they see outside — sunshine, passing clouds — to how much energy is currently being generated or will be available later in the day. Their study deployed a set of **technology probes** to surface these relationships and provoke reflection. Two of those probes are the basis of what you build here:

- **The Dial** — a display of current solar generation.
- **The Energy Forecast** — a 24-hour timeline of expected generation derived from weather data.

You begin by building the forecast panel as a **simulated prototype** using weather data — no hardware required. Once that simulation has been tested with a participant, you add a **live light sensor** (a solar proxy) that makes the current-generation panel real. The final probe is a hybrid: simulated forecast alongside live sensing, deployed as a hosted web interface.

By the end of this tutorial you will have:

- A simulated forecast panel fed by weather data and hosted on Data Foundry.
- Experience running a simulated probe walkthrough session and reflecting on what it tested and left untested.
- A live light sensor (Connected Interaction Kit) publishing a generation index to Data Foundry via OOCSI.
- A hybrid probe interface showing both current and forecast generation, shareable as a URL.
- A grounded understanding of when to choose a local vs. cloud pipeline and what that trade-off means for participant privacy.

## Before you start

### Hardware you need

- Connected Interaction Kit with a **BitsyExpander**
- **TSL2591** high-dynamic-range light sensor (I²C, Grove connector)
- USB cable for power and programming
- Access to a WiFi network (`TUD-facility` on campus)

The hardware is only needed for Part B. You can complete Part A without it.

### Set up your Data Foundry project

1. Open (or create) a project in [Data Foundry](https://datafoundry.io.tudelft.nl/).
2. In the **General** tab, add an **IoT Dataset**. Name it `Solar Probe`. Note its numeric dataset ID.
3. In the **Sources** tab, add a device. Call it `light-sensor-01`. Copy its device ID.
4. On the **Solar Probe** IoT Dataset page, go to **Configuration → Data from OOCSI** and enter `probe/solar/generation` as the topic name. Click **Save**.
5. Back in the **General** tab, add an **Existing Dataset**. Name it `Probe Interface`. This is where the HTML interface files will live.

## Part A — The Simulated Probe

The simulated prototype answers the first design question: *does this concept resonate with participants?* You build and deploy the interface before acquiring any live sensor data.

---

### Step 1 — Prepare your weather data

The Energy Forecast panel needs a source of solar irradiance data. Two options are available, each with different trade-offs.

#### Option A: Static CSV

Download [weather_sample.csv](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.2/weather_sample.csv) from the tutorial resources. This synthetic file contains one week of hourly data:

| Column | Description |
|---|---|
| `timestamp` | ISO 8601 datetime |
| `cloud_cover` | 0–100 % |
| `uv_index` | 0–11 |
| `generation_index` | 0–100, derived as `max(0, (1 − cloud_cover/100) × uv_index/11 × 100)` |

Upload the CSV to your **Probe Interface** Existing Dataset in Data Foundry. Under **Configuration**, activate **Web Access** — Data Foundry provides a direct URL to the raw file that your HTML page can fetch.

#### Option B: Live Open-Meteo API

[Open-Meteo](https://open-meteo.com/) provides free hourly weather forecasts with no API key. The JavaScript snippet below retrieves the next 24 hours of cloud cover and UV index:

```javascript
const LAT = 52.00; // replace with your latitude
const LON = 4.36;  // replace with your longitude

const url = `https://api.open-meteo.com/v1/forecast` +
            `?latitude=${LAT}&longitude=${LON}` +
            `&hourly=cloud_cover,uv_index&forecast_days=1&timezone=auto`;

const response = await fetch(url);
const data     = await response.json();
// data.hourly.time, data.hourly.cloud_cover, data.hourly.uv_index
```

Apply the same formula to derive `generation_index` for each hour in JavaScript.

#### Choosing between the options

| | Option A — Static CSV | Option B — Live API |
|---|---|---|
| Works offline | Yes | No |
| Reproducible across sessions | Yes | No — weather changes |
| Reflects a real forecast | No | Yes |
| Requires API key | No | No |
| Sends location to external server | No | Yes (lat/lon to Open-Meteo) |
| Best for | First pilot sessions, reliable demos | Later sessions where realism matters |

{: .note }
> **Location as personal data.** The Open-Meteo API call sends the participant's approximate home location to an external server. For home deployments, check whether this transmission is covered by your study's consent documentation. The static CSV avoids the issue entirely.

---

### Step 2 — Build and host the forecast panel

The tutorial resources include a starter interface — [probe-forecast.html](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.2/probe-forecast.html) — that renders the generation index as a 24-hour bar chart. Open it locally in a browser first to confirm it displays correctly with your chosen data source.

To extend or customise the panel, describe what you want to a code-capable LLM and provide the data schema rather than any actual participant data:

```
I am building a technology probe for domestic solar energy microgeneration.
Data: hourly timestamps and a generation_index (0–100).
Build a single-page HTML interface with:
- A 24-hour bar chart of generation_index, coloured from light grey (low) to amber (high)
- The current hour's bar highlighted with a border
- A one-line summary: "Peak generation expected at [time] — index [value]"
Use no external dependencies — vanilla JS and inline CSS only.
```

Once your interface is ready:

1. Upload `probe-forecast.html` to your **Probe Interface** Existing Dataset in Data Foundry.
2. Go to **Configuration → Web Access** and activate public access.
3. Open the URL in a browser and confirm the chart loads correctly.

{: .note }
> For the static CSV option, the page will fetch the file from the Data Foundry URL you configured in Step 1. For the live API option, it will fetch from Open-Meteo on load. Both result in the same 24-hour bar chart — only the data provenance differs.

---

### Step 3 — Run a simulated probe session

Present the hosted URL to a colleague or participant on a tablet or laptop, framing it as a working prototype of an installed product. Your role is to observe, not to explain the interface. Note:

- What does the participant look at first — the current hour or the full-day arc?
- Do they spontaneously connect the forecast to a household task ("I should run the washing machine at noon")?
- What questions does the interface leave unanswered?
- What do they think the probe *cannot* tell them?

After 10–15 minutes, debrief: explain that the data is pre-recorded or simulated, and discuss what a live version would add.

**Reflect before continuing.** What did the simulation test? What did it leave untestable? Did participants engage with the forecast, or did they want to know what was happening *right now*? If your participants asked about the current moment, that is your motivation for Part B.

## Part B — The Live Probe

Part B makes the "right now" panel real. A light sensor on the Connected Interaction Kit measures ambient light as a proxy for solar generation and publishes it to Data Foundry in real time.

---

### Step 4 — Wire up the light sensor and publish data

#### Hardware

Connect the TSL2591 sensor to one of the I²C Grove ports on the BitsyExpander. No pin mapping is needed — the Grove connector handles power and data lines.

#### Firmware

The full firmware is in the [tutorial resources](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.2/esp32/). Configure `settings.py`:

```python
settings = {
    "ssid":                 "TUD-facility",
    "password":             "your-wifi-password",
    "device_id":            "your-device-id",       # from Data Foundry Sources tab
    "oocsi_base":           "oocsi.id.tue.nl",
    "oocsi_topic":          "probe/solar/generation",
    "collection_frequency": 30                      # seconds between readings
}
```

The core sensing loop in `code.py`:

```python
import adafruit_tsl2591

sensor = adafruit_tsl2591.TSL2591(i2c)

while True:
    lux = sensor.lux or 0
    # Normalise to a 0–100 generation index.
    # 80,000 lux ≈ bright summer sun at this latitude; adjust if needed.
    max_lux = 80000
    generation_index = min(100, int(lux / max_lux * 100))

    oocsi.send(settings["oocsi_topic"], {
        "device_id":        settings["device_id"],
        "lux":              lux,
        "generation_index": generation_index
    })
    time.sleep(settings["collection_frequency"])
```

Deploy to the board following the same procedure as T7.1 (copy to `CIRCUITPY`, watch the serial monitor for a successful OOCSI connection).

{: .note }
> **Generation index as a design decision.** You are mapping lux to a 0–100 index rather than measuring watts per square metre. This is intentional — the probe is testing whether the concept of relative solar availability resonates with participants, not delivering a calibrated energy reading. The "When Looking Out of the Window" paper used the same approach: their Dial showed current generation without unit labels, which deliberately provoked participants to ask "but how much is that, exactly?" That question is a finding.

#### Verify data in Data Foundry

Open the **Solar Probe** IoT Dataset and click **Data Table**. Rows should appear every 30 seconds containing `lux` and `generation_index`. Cover the sensor with your hand and confirm `generation_index` drops toward zero. Place it near a bright window and confirm it rises.

---

### Step 5 — Update the interface with live readings

Now connect the live sensor data to the probe interface. Extend `probe-forecast.html` to also fetch the latest reading from your IoT Dataset using the Data Foundry JavaScript client.

Add the following to your HTML file:

```html
<script src="https://data-foundry.net/js/datafoundry.min.js"></script>
<script>
  const PROJECT_ID = 'your-project-id';  // from Data Foundry project settings
  const DATASET_ID = 'your-dataset-id';  // numeric ID of Solar Probe IoT Dataset

  async function updateCurrentGeneration() {
    const df     = new DataFoundry(PROJECT_ID);
    const latest = await df.dataset(DATASET_ID).latest();
    const index  = latest?.generation_index ?? 0;

    document.getElementById('current-index').textContent = index;
    document.getElementById('current-bar').style.width   = index + '%';
  }

  updateCurrentGeneration();
  setInterval(updateCurrentGeneration, 60000); // refresh every 60 s
</script>
```

Add a "Now" panel above the 24-hour forecast chart — a large number or filled bar showing the current generation index. When you reload the page, it should display the most recent reading from the sensor.

Re-upload `probe-forecast.html` to the **Probe Interface** Existing Dataset (overwriting the previous version). Open the URL — the probe now combines a live current-generation reading with the simulated or API-based forecast.

---

### Step 6 — Deploy the hybrid probe

Place the Connected Interaction Kit near a window that receives direct sunlight at some point during the day. Leave it running. Share the probe URL with a participant on a device they keep visible — tablet, phone, or a small dedicated screen.

**Deployment checklist** (from the textbook):

- [ ] Sensor returns expected values across the full range (full sun to overcast to indoors)
- [ ] Data appears in the IoT Dataset without gaps after a power interruption
- [ ] Timestamp synchronisation confirmed (OOCSI applies server-side timestamps, so no NTP step is needed here)
- [ ] Participant has been briefed and consent documented
- [ ] A data retrieval plan is in place

After one or more days of deployment, compare the live generation index time-series in Data Foundry against the forecast values shown in the interface. Did the forecast match reality? A systematic divergence — the forecast consistently overestimating on cloudy days — is itself a design finding: it tells you something about how much participants can rely on the probe for task planning.

## Alternative: keeping data local with Raspberry Pi and MQTT

The main path above routes sensor readings through Data Foundry's cloud infrastructure. This is convenient and accessible from anywhere, but means that every reading — and the approximate home location sent to Open-Meteo — leaves the home network. For deployments where participant privacy is a primary concern, a fully local alternative replaces the cloud components:

| Role | Local tool | Replaces |
|---|---|---|
| Message routing | Mosquitto (MQTT broker) on Raspberry Pi | OOCSI relay |
| Time-series storage | InfluxDB on Raspberry Pi | Data Foundry IoT Dataset |
| Live dashboard | Grafana on Raspberry Pi | Data Foundry JS client |

The ESP32 firmware changes one line — replacing the OOCSI send with an MQTT publish to the Raspberry Pi's local IP address. Mosquitto routes the message; Telegraf (InfluxData's data agent) subscribes and writes to InfluxDB; Grafana serves the live dashboard on the local network without any data leaving the home.

```
ESP32 → WiFi/MQTT → Mosquitto (Pi) → Telegraf → InfluxDB → Grafana
```

The textbook's home sensor study used exactly this architecture: SensorTag devices transmitted wirelessly to a Raspberry Pi hub, InfluxDB stored the time series, and Grafana served visualisations over the local network. Data never left the home unless participants actively chose to share it.

The Raspberry Pi can also replace the Open-Meteo API call: a Python script running on the Pi fetches the forecast once per hour and writes it to a local file served to the probe interface. This is edge processing — the API call originates from inside the home, but the forecast data never passes through a cloud data store.

The trade-off is access: local data is private and offline-capable but inaccessible for remote monitoring or cross-participant analysis without an explicit sharing step.

{: .note }
> [Node-RED](https://nodered.org) is a visual flow-programming environment that runs on a Raspberry Pi. It subscribes to MQTT topics, transforms data, writes to InfluxDB, and triggers notifications by connecting graphical nodes — without writing traditional code. For designers who want to validate pipeline logic quickly before committing to Python, Node-RED is often the fastest path from sensor to dashboard.

## Reflect

Before drawing conclusions from the probe deployment, consider:

1. **Simulation vs. live.** What did the simulated probe session in Part A test that a paper prototype could not? What did the live sensor in Part B reveal that the simulation could not? Did the two phases surface different concerns from participants?

2. **The generation index as a design decision.** You chose a simple lux-to-index proxy rather than a calibrated watt output. What does that choice communicate to participants? The "When Looking Out of the Window" paper found that participants wanted to know whether they had *enough* energy to run a specific appliance. Could your 0–100 index answer that question? What would you need to add?

3. **Forecast accuracy and trust.** If the forecast diverged significantly from the live readings — clouds arrived earlier than predicted, or a sunny afternoon was forecast but overcast — how might that affect a participant's willingness to act on the probe's recommendations? How would you detect and communicate forecast uncertainty in the interface?

4. **Data that leaves the home.** The Open-Meteo API call sends a location to an external server. Sending sensor readings to Data Foundry sends energy-behaviour data to a cloud service. Neither was explicitly mentioned in the probe walkthrough in Step 3. Should participants know about these data flows? How would you explain them in plain language?

5. **What the probe does not sense.** This probe measures generation only — not consumption. What questions can it therefore not answer? Is that an implementation shortcut or a deliberate design decision? The "When Looking Out of the Window" paper treated it as a deliberate choice: starting with generation focused participants on the supply side and made the gap between production and consumption a discovery, not a given.
