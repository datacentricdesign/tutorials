## Introduction

A connected product that senses human behaviour generates continuous streams of numbers — one row per second or minute, stretching across days and weeks. Before drawing conclusions from this data you need to get familiar with it: its shape, its gaps, its rhythms, and its surprises. This tutorial practises that process end-to-end on a real human-activity dataset.

The data comes from the **ExtraSensory** study, in which 60 participants in the United States carried a smartphone and wore a smartwatch for seven consecutive days while going about their daily lives. Every 60 seconds the phone extracted features from its sensors and recorded whatever activity the participant was doing at that moment. You will work with data from five of those participants, already selected and pre-processed for this tutorial.

The primary signal you will analyse is `accel_std` — the standard deviation of the phone accelerometer magnitude within each 60-second window. At rest the accelerometer reads only gravity, so `accel_std` is near zero. Walking produces rhythmic footstep vibrations, raising it to around 0.15 g. This simple proxy for movement intensity is enough to reveal daily routines, rest periods, and unusual events.

By the end you will have:
- Loaded and profiled a multi-day, multi-participant IoT dataset
- Applied smoothing at different scales to reveal patterns at different resolutions
- Built an hour-of-day × day-of-week heatmap that summarises a participant's weekly rhythm
- Detected anomalies and interpreted them using ground-truth activity labels
- Produced a clean visualisation ready to use in a Participatory Data Analysis session

## Step 1 — Load and profile the data

Open your notebook and delete the default JavaScript cell.

### Load the CSV

```python
from pyodide.http import pyfetch
import pandas as pd
import io

# Paste the URL provided by your instructor here
DATASET_URL = "paste-your-dataset-url-here"

resp = await pyfetch(DATASET_URL)
data = await resp.string()
df   = pd.read_csv(io.StringIO(data))

print("Shape:", df.shape)
print("Columns:", df.columns.tolist())
df.head()
```

You should see roughly 45 000 rows and 18 columns. The key columns are:

| Column | Type | Description |
|---|---|---|
| `participant` | string | Anonymised participant ID (`P01`–`P05`) |
| `timestamp` | string | ISO-8601 datetime (UTC) of the 60-s window |
| `accel_mean` | float | Mean phone acceleration magnitude (g) |
| `accel_std` | float | Std of phone acceleration magnitude (g) — proxy for movement intensity |
| `watch_accel_mean` | float | Mean watch acceleration magnitude |
| `gps_updates` | int | Number of valid GPS fixes in the window |
| `battery_level` | float | Phone battery fraction 0–1 |
| `label_*` | int / NaN | Ground-truth activity labels: `1` = yes, `0` = no, `NaN` = not labeled |

### Parse timestamps

```python
df["ts"] = pd.to_datetime(df["timestamp"], utc=True)
df = df.sort_values(["participant", "ts"]).reset_index(drop=True)

print("Coverage per participant:\n")
print(f"{'ID':>4}  {'Start':>12}  {'End':>12}  {'Rows':>6}  {'Days':>5}")
for p, grp in df.groupby("participant"):
    days = (grp["ts"].max() - grp["ts"].min()).days + 1
    print(f"{p:>4}  {str(grp['ts'].min().date()):>12}  "
          f"{str(grp['ts'].max().date()):>12}  {len(grp):>6,}  {days:>5}")
```

Expected output (dates will vary by participant):

```
Coverage per participant:

  ID         Start           End    Rows   Days
 P01    2015-10-07    2015-10-14   9,761      7
 P02    2015-11-30    2015-12-07   7,521      7
 P03    2015-11-30    2015-12-07   9,686      7
 P04    2015-12-07    2015-12-14   9,539      7
 P05    2015-11-23    2015-11-30   9,242      7
```

Each participant contributed approximately 7 days of data. The participants were recorded at different calendar weeks in late 2015.

### Profile the sensor columns

```python
sensor_cols = ["accel_mean", "accel_std", "watch_accel_mean",
               "gps_updates", "battery_level"]
print(df[sensor_cols].describe().round(4))
```

Look at the output carefully. Two things should stand out:

1. **`accel_mean` is around 1.0**, while **`watch_accel_mean` is around 1 000**. Both columns measure accelerometer magnitude, but they use different units — one is in g-units, the other in milli-g. You will need to account for this before using the watch data. For this tutorial we will focus on the phone signal, but this is the kind of unit inconsistency that real-world sensor datasets regularly contain.

2. **`gps_updates` and `battery_level` have missing values.** Run `df[sensor_cols].isna().sum()` to see how many. Missing sensor readings can mean the feature was unavailable (GPS indoors), or the sensor was off, or the phone was not worn.

### Check label coverage

```python
label_cols = [c for c in df.columns if c.startswith("label_")]

print("Label coverage — fraction of rows where label is known (not NaN):\n")
coverage = df[label_cols].notna().mean().sort_values(ascending=False)
for col, frac in coverage.items():
    bar = "█" * int(frac * 30)
    print(f"  {col:<22} {bar:<30} {frac:.0%}")
```

You will notice large differences: `label_at_home` and `label_lying` are known for nearly every row, while `label_cycling` and `label_running` are mostly `NaN`. This does not mean the participant was never running — it means those activities were **not reported** during those windows. In the ExtraSensory study, participants used a phone app to label their current activity. Rare activities were simply selected less often.

This is a common pattern in opportunistic labelling studies: **`NaN` in a label column is not the same as `0`**. A `NaN` means "we don't know"; a `0` means "the participant confirmed this was not happening".

{: .note }
> **Sampling regularity check.** Before any time-series analysis, check whether your data arrives at regular intervals. Run `df[df.participant == "P02"]["ts"].diff().dt.total_seconds().div(60).describe().round(1)` to see the gap distribution. You should see a median of ~1 minute, but also some large gaps — periods where data is simply absent. These are not `NaN` values; they are missing rows entirely. Keep this distinction in mind.

## Step 3 — Smooth the signal

The raw 1-minute signal is spiky. Smoothing replaces each point with the average of its neighbours, trading detail for readability. The key choice is the **window size**.

```python
fig, ax = plt.subplots(figsize=(14, 3))

# Raw signal in light grey as background
ax.plot(p02.index, p02["accel_std"], lw=0.3, color="lightgray", label="raw (1 min)")

# Three window sizes — each reveals a different time scale
for label, color, lw in [
    ("5 min",  "steelblue",   0.6),
    ("30 min", "darkorange",  1.0),
    ("2 h",    "crimson",     1.5),
]:
    smoothed = p02["accel_std"].rolling(label, center=True).mean()
    ax.plot(p02.index, smoothed, lw=lw, color=color, label=f"rolling mean {label}")

ax.legend(loc="upper right", fontsize=8)
ax.set_ylabel("accel_std (g)")
ax.set_title("P02 — effect of smoothing window on signal")
ax.xaxis.set_major_formatter(mdates.DateFormatter("%b %d"))
plt.xticks(rotation=30)
plt.tight_layout()
plt.show()
```

Experiment by zooming into a single day to compare the window sizes more clearly:

```python
# Zoom in to one day — replace with a date that appears active in your plot
day_start = "2015-12-01"
day_end   = "2015-12-02"
p02_day = p02[day_start:day_end]

fig, ax = plt.subplots(figsize=(14, 3))
ax.plot(p02_day.index, p02_day["accel_std"],
        lw=0.5, color="lightgray", label="raw")
for label, color, lw in [("5 min", "steelblue", 0.8), ("30 min", "darkorange", 1.2)]:
    ax.plot(p02_day.index, p02_day["accel_std"].rolling(label, center=True).mean(),
            lw=lw, color=color, label=f"rolling mean {label}")

ax.legend()
ax.set_ylabel("accel_std (g)")
ax.xaxis.set_major_formatter(mdates.DateFormatter("%H:%M"))
plt.xticks(rotation=30)
ax.set_title(f"P02 — one day zoomed in ({day_start})")
plt.tight_layout()
plt.show()
```

Reflect on what you observe:
- At the **5-minute window**, can you still see individual activity bouts (short walks, a trip to the kitchen)?
- At the **30-minute window**, what remains visible, and what is lost?
- At **2 hours**, the curve describes only broad morning/afternoon/evening transitions. When would this level of detail be useful in a design project?

There is no correct window size — the right choice depends on the question you are asking. If you are designing a real-time activity detection algorithm, you need the fine-grained signal. If you are designing a weekly feedback summary for a health app, a 2-hour smoothing may be perfectly adequate.

## Step 4 — Aggregate to a heatmap

Individual days are noisy and hard to compare. Aggregating across the same hour on the same weekday across multiple weeks reveals stable rhythms. This is the **hour-of-day × day-of-week** heatmap — one of the most useful views for human-activity data.

### Build the pivot table

```python
# Resample to hourly means first, then compute weekday/hour averages
p02_hourly = p02["accel_std"].resample("1h").mean()

# Build a DataFrame with hour and weekday columns for pivoting
p02_hourly_df = p02_hourly.to_frame()
p02_hourly_df["hour"]    = p02_hourly_df.index.hour
p02_hourly_df["weekday"] = p02_hourly_df.index.day_name()

pivot = (
    p02_hourly_df
    .groupby(["weekday", "hour"])["accel_std"]
    .mean()
    .unstack("hour")
)

# Sort days Monday → Sunday
day_order = ["Monday", "Tuesday", "Wednesday", "Thursday",
             "Friday", "Saturday", "Sunday"]
pivot = pivot.reindex([d for d in day_order if d in pivot.index])

print(pivot.round(3))
```

### Plot the heatmap

```python
fig, ax = plt.subplots(figsize=(14, 4))

im = ax.imshow(
    pivot.values,
    aspect="auto",
    cmap="YlOrRd",
    origin="upper",
    vmin=0, vmax=0.08,       # set a common scale so participants are comparable
)

ax.set_xticks(range(0, 24, 2))
ax.set_xticklabels([f"{h:02d}:00" for h in range(0, 24, 2)], fontsize=8)
ax.set_yticks(range(len(pivot)))
ax.set_yticklabels(pivot.index, fontsize=9)
ax.set_xlabel("Hour of day")
ax.set_title("P02 — mean activity level (accel_std) by hour and day of week")
plt.colorbar(im, ax=ax, label="accel_std (g)", fraction=0.02, pad=0.02)
plt.tight_layout()
plt.show()
```

### Compare two participants

```python
def activity_heatmap(df_full, participant):
    """Return a (weekday × hour) pivot of mean accel_std for one participant."""
    part = df_full[df_full["participant"] == participant].set_index("ts").sort_index()
    hourly = part["accel_std"].resample("1h").mean().to_frame()
    hourly["hour"]    = hourly.index.hour
    hourly["weekday"] = hourly.index.day_name()
    pivot = (
        hourly.groupby(["weekday", "hour"])["accel_std"]
        .mean()
        .unstack("hour")
    )
    day_order = ["Monday", "Tuesday", "Wednesday", "Thursday",
                 "Friday", "Saturday", "Sunday"]
    return pivot.reindex([d for d in day_order if d in pivot.index])


fig, axes = plt.subplots(1, 2, figsize=(16, 4))

for ax, participant in zip(axes, ["P01", "P02"]):
    pivot = activity_heatmap(df, participant)
    im = ax.imshow(pivot.values, aspect="auto", cmap="YlOrRd",
                   origin="upper", vmin=0, vmax=0.08)
    ax.set_xticks(range(0, 24, 3))
    ax.set_xticklabels([f"{h:02d}:00" for h in range(0, 24, 3)], fontsize=8)
    ax.set_yticks(range(len(pivot)))
    ax.set_yticklabels(pivot.index, fontsize=8)
    ax.set_title(participant)

plt.colorbar(im, ax=axes, label="accel_std (g)", fraction=0.02, pad=0.02)
plt.suptitle("Activity rhythm — hour of day × day of week", fontsize=11)
plt.tight_layout()
plt.show()
```

Look at the two heatmaps side by side:
- Do both participants share a similar broad structure (active during the day, quiet at night)?
- Where do they differ? Are some weekdays structurally different from others for one participant but not the other?
- What does a blank cell (NaN, shown as white or grey) tell you about data coverage?

{: .note }
> Seven days of data means at most one observation per weekday cell. Some cells may contain only one or two hourly samples. Treat this heatmap as an exploratory sketch, not a statistical estimate.

## Step 5 — Detect anomalies

Anomalies are moments where the signal departs from what the surrounding context would predict. A simple and interpretable method is to flag points that exceed **2 standard deviations above a rolling baseline**. This adapts to the participant's changing activity level throughout the day.

### Compute the rolling threshold

```python
# 30-minute rolling window, centred on each point
roll_mean = p02["accel_std"].rolling("30min", center=True, min_periods=10).mean()
roll_std  = p02["accel_std"].rolling("30min", center=True, min_periods=10).std()

threshold = roll_mean + 2 * roll_std
anomalies = p02["accel_std"] > threshold

print(f"Anomaly rows: {anomalies.sum()} ({100 * anomalies.mean():.1f}% of dataset)")
```

You should see roughly 150–200 flagged rows — around 2% of the total. 

### Visualise a 3-day window

```python
# Replace with a date range that appeared active in your Step 2 plot
window_start = "2015-12-01"
window_end   = "2015-12-04"

p02_w    = p02[window_start:window_end]
thresh_w = threshold[window_start:window_end]
anom_w   = anomalies[window_start:window_end]

fig, ax = plt.subplots(figsize=(14, 3))
ax.plot(p02_w.index, p02_w["accel_std"],
        lw=0.5, color="steelblue", label="signal")
ax.plot(p02_w.index, thresh_w,
        lw=1.0, color="darkorange", ls="--", label="2σ threshold")
ax.scatter(p02_w.index[anom_w], p02_w["accel_std"][anom_w],
           color="crimson", s=12, zorder=5, label="anomaly")
ax.legend(fontsize=8)
ax.set_ylabel("accel_std (g)")
ax.set_title("P02 — anomalies (3-day view)")
ax.xaxis.set_major_formatter(mdates.DateFormatter("%b %d %H:%M"))
plt.xticks(rotation=30)
plt.tight_layout()
plt.show()
```

### Interpret using ground-truth labels

Detecting an anomaly is only the first step. The more important question is: **what do the anomalies actually represent?** This is where the ExtraSensory activity labels become useful.

```python
label_cols = [c for c in p02.columns if c.startswith("label_")]
anom_rows  = p02[anomalies]

print("Activity labels at anomaly points (only labels with at least one match):\n")
print(f"  {'Label':<25} {'Anomaly rows with label=1':>25}  {'% of all rows with label=1':>28}")
print("  " + "-" * 82)
for col in label_cols:
    n_anom  = int((anom_rows[col] == 1).sum())
    n_total = int((p02[col] == 1).sum())
    if n_anom > 0:
        pct = 100 * n_anom / max(n_total, 1)
        print(f"  {col:<25} {n_anom:>25}  {pct:>27.1f}%")
```

You should see that anomaly rows include a mix of labels. Anomalies during `label_sitting = 1` are probably brief moments of phone movement (checking the phone, placing it down). Anomalies during `label_walking = 1` likely correspond to running or stair-climbing. A few anomalies during `label_sleeping = 1` are puzzling — what might explain high phone movement during sleep?

This is the interpretive question at the heart of EDA in design research. The algorithm flags something as unusual; the domain context (and ultimately the participant) explains what it means.

## Step 6 — Prepare for Participatory Data Analysis

You have now explored the data from a researcher's perspective. The next step in data-centric design is to bring these findings back to the people the data is about — a method called **Participatory Data Analysis (PDA)**. The goal is not to report conclusions, but to show participants a visualisation of their data and invite their interpretation.

A good PDA stimulus is:
- **Legible** to someone with no data background — no axis labels like "accel_std", no jargon
- **Surprising enough** to provoke conversation — something the participant did not know or expect
- **Open-ended** — it poses a question, not a conclusion

The heatmap from Step 4 is a good candidate. Produce a clean, participant-facing version:

```python
fig, ax = plt.subplots(figsize=(12, 4))

pivot = activity_heatmap(df, "P02")

im = ax.imshow(pivot.values, aspect="auto", cmap="YlOrRd",
               origin="upper", vmin=0, vmax=0.08)

ax.set_xticks(range(0, 24, 2))
ax.set_xticklabels(
    [f"{h:02d}:00" for h in range(0, 24, 2)],
    fontsize=9
)
ax.set_yticks(range(len(pivot)))
ax.set_yticklabels(pivot.index, fontsize=10)
ax.set_xlabel("Time of day", fontsize=11)
ax.set_title("Your movement intensity — last week", fontsize=13, pad=12)
plt.colorbar(im, ax=ax, label="Movement intensity", fraction=0.02, pad=0.02)
plt.tight_layout()
plt.savefig("pda_stimulus.png", dpi=150, bbox_inches="tight")
plt.show()

print("Saved as pda_stimulus.png")
```

Notice the changes from the exploratory version: the title is in plain language ("Your movement intensity"), the colour bar label drops units, and the y-axis shows days of the week rather than technical codes.

Now, before showing this to a participant, write down **three questions** you would use to open the conversation. Good PDA questions:
- Start with the participant's experience: *"Does this pattern look familiar to you?"*
- Invite interpretation of a specific feature: *"This Thursday evening looks much more active than the others — what were you doing?"*
- Surface what the data cannot explain: *"There are a few very quiet periods during what should be your daytime. What might those represent?"*

Write your three questions in a new text cell in the notebook before proceeding.

## Reflect

Before using this approach in a design project, consider:

1. **From proxy to meaning.** `accel_std` measures phone movement, not human activity. When you interpret a low-activity period as "sleep", what assumptions are you making? When might those assumptions be wrong — and how would a connected product that acted on this signal fail?

2. **Scale as a design choice.** You smoothed the signal at 5 minutes, 30 minutes, and 2 hours. Each scale reveals different design-relevant phenomena. If you were designing a prompt that nudged users to move more, which scale would you use to detect the trigger moment? If you were building a weekly review feature, which scale would you use to characterise the week?

3. **The meaning of missing data.** You saw two types of absence: `NaN` values in label columns (the participant did not report this activity), and entire missing rows (no data was collected). These are different problems requiring different responses. If a connected product showed users their "active time" and missed an 8-hour gap, how should it handle the missing period — fill, flag, or stay silent?

4. **Anomalies as hypotheses.** The anomaly detector flagged moments of high phone movement. Some correspond to walking; some to sitting. Before building a product that reacts to anomalies, what would you need to know about false-positive rate — and how would you find out without access to ground-truth labels in a real deployment?

5. **From exploration to PDA.** You produced a heatmap from the researcher's side. Before showing it to a participant, what would you need to know about their data literacy? What if they find the heatmap confusing rather than illuminating? How would you adjust the representation?
