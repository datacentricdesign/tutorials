# T4.3 · Exploring Text Data

**Textbook context:** Text data as design material; RTA phases 1–3 (familiarisation, initial coding, candidate themes); the tension between predefined schemas and inductive interpretation.

**Tools:** Starboard Notebook (corpus profiling), Atlas.ti (qualitative analysis)

## Introduction

Text data carries something sensors cannot: the words people chose to describe their experience. But working with text requires a different kind of discipline. The temptation is to count words before reading them, to run algorithms before understanding what the corpus contains. This tutorial builds the habit of the opposite: read first, compute later.

The corpus is a set of ethics assessments written by approximately 100 master design students. Each student analysed a specific connected product — an electric scooter, a smart cabin, a connected wheelchair, a hotel room sensor — and wrote a response for each of six ethics dimensions: informed consent, privacy, information security, physical safety, trust, and sustainability. The structure looks like an analysis: six neatly labelled columns, one row per student. But the structure is a starting point, not a finding. Whether those six dimensions actually capture what the students were thinking about — or whether they fragment concerns that belong together — is precisely what exploration will tell you.

You will work in two tools. **Starboard** handles the profiling step: loading the dataset, understanding its shape, and examining the sub-corpus you will analyse in depth. **Atlas.ti** handles the qualitative work: reading, coding, and constructing candidate themes using Reflexive Thematic Analysis phases 1–3.

By the end you will have:
- A profile of the full corpus (item counts, text lengths, response patterns) that tells you what you have before you read it
- A set of initial codes grounded in close reading of 13 wheelchair ethics assessments
- 3–5 candidate themes with one-sentence definitions — including a note on which themes cut across the predefined column structure
- A written prediction to carry into T5.3

## Step 1 — Load and profile the corpus

Open your Starboard notebook and delete the default JavaScript cell. Paste your dataset URL where indicated.

### Load the dataset

```python
from pyodide.http import pyfetch
import pandas as pd
import io

DATASET_URL = "https://raw.githubusercontent.com/datacentricdesign/tutorials/refs/heads/main/resources/T4.3/data-design-ethics.csv"

resp = await pyfetch(DATASET_URL)
data = await resp.string()
df = pd.read_csv(io.StringIO(data))

print("Shape:", df.shape)
print("Columns:\n", "\n".join(f"  {c}" for c in df.columns))
df.head(3)
```

You should see 107 rows and 10 columns. The first two columns identify each respondent (`Name`) and their assigned product (`Product`). The remaining eight columns hold free-text responses, one per ethics dimension.

### Who wrote about what?

```python
import matplotlib.pyplot as plt

product_counts = df["Product"].value_counts()

fig, ax = plt.subplots(figsize=(8, 3.5))
bars = ax.barh(product_counts.index, product_counts.values, color="steelblue")
ax.bar_label(bars, padding=4, fontsize=9)
ax.set_xlabel("Number of respondents")
ax.set_title("Corpus composition — respondents per product")
ax.invert_yaxis()
plt.tight_layout()
plt.show()
```

Note the spread: between 9 and 19 students per product. This matters for T5.3 — unequal group sizes can distort computational results.

### How much did people write?

```python
# Define the six main ethics dimensions
ETHICS_COLS = [
    "Ethics concern regarding Informed consent",
    "Ethics concern regarding Privacy",
    "Ethics concern regarding Information security",
    "Ethics concern regarding Physical safety",
    "Ethics concern regarding Trust",
    "Ethics concern regarding Sustainability",
]

SHORT = {c: c.replace("Ethics concern regarding ", "").strip() for c in ETHICS_COLS}

# Compute text length per cell
lengths = df[ETHICS_COLS].map(lambda x: len(str(x)) if pd.notna(x) else 0)
lengths.columns = [SHORT[c] for c in ETHICS_COLS]

fig, ax = plt.subplots(figsize=(10, 4))
lengths.boxplot(ax=ax, vert=True, patch_artist=True,
                boxprops=dict(facecolor="steelblue", alpha=0.5))
ax.set_ylabel("Response length (characters)")
ax.set_title("Distribution of response length per ethics dimension")
plt.xticks(rotation=20, ha="right")
plt.tight_layout()
plt.show()

print("Median response length per dimension:")
print(lengths.median().round(0).to_string())
```

Look at the spread within each box: some dimensions have a wide range, meaning some students wrote extensively and others minimally. A short response is not necessarily a thin one — brevity can be precise — but very short responses (under 100 characters) are often placeholder text rather than substantive analysis.

### Which product × dimension combinations are richest?

```python
import numpy as np

# Median length per product × dimension
pivot = (
    df.assign(**{SHORT[c]: df[c].map(lambda x: len(str(x)) if pd.notna(x) else 0) for c in ETHICS_COLS})
    .groupby("Product")[[SHORT[c] for c in ETHICS_COLS]]
    .median()
)

fig, ax = plt.subplots(figsize=(10, 4))
im = ax.imshow(pivot.values, aspect="auto", cmap="YlGnBu")
ax.set_xticks(range(len(pivot.columns)))
ax.set_xticklabels(pivot.columns, rotation=25, ha="right", fontsize=9)
ax.set_yticks(range(len(pivot.index)))
ax.set_yticklabels(pivot.index, fontsize=9)
ax.set_title("Median response length (chars) — product × ethics dimension")
plt.colorbar(im, ax=ax, label="Median characters", fraction=0.03, pad=0.03)
plt.tight_layout()
plt.show()
```

The heatmap reveals which combinations produced the most substantive responses — not better or worse, but more or less expansive. Notice where the Permobil connected wheelchair row sits relative to others.

## Step 2 — Examine the wheelchair sub-corpus

The rest of this tutorial works with the wheelchair responses. With 13 respondents it is small enough to read in full — a crucial advantage for phases 1–3 of RTA, where close reading is the method.

### Response lengths per respondent

```python
wc = df[df["Product"] == "Permobil connected wheelchair"].copy()
wc_lengths = wc[ETHICS_COLS].map(lambda x: len(str(x)) if pd.notna(x) else 0)
wc_lengths.index = wc["Name"].values
wc_lengths.columns = [SHORT[c] for c in ETHICS_COLS]

fig, ax = plt.subplots(figsize=(10, 4))

ax.set_xticklabels(wc_lengths.columns, rotation=20, ha="right", fontsize=9)
ax.set_yticks(range(len(wc_lengths.index)))

ax.set_title("Wheelchair sub-corpus — response length per respondent × dimension")
plt.colorbar(im, ax=ax, label="Characters", fraction=0.03, pad=0.03)

```

This is a first look at your corpus without reading it. Notice: who wrote most? Which dimensions received the most attention? Are there any blank cells (length ≈ 0)?

### A first taste

Print two complete responses — one brief, one extensive — to calibrate what you are about to read.

```python
sorted_by_length = wc.assign(
    total=wc[ETHICS_COLS].map(lambda x: len(str(x)) if pd.notna(x) else 0).sum(axis=1)
).sort_values("total")

for label, row in [("Shortest respondent", sorted_by_length.iloc[0]),
                   ("Longest respondent",  sorted_by_length.iloc[-1])]:
    print(f"\n{'='*60}")
    print(f"{label} — {row['Name']}")
    print(f"{'='*60}")
    for col in ETHICS_COLS[:3]:   # first three dimensions only
        print(f"\n--- {SHORT[col]} ---")
        print(str(row[col])[:500])
```

Read these two excerpts carefully. You are not yet coding — just noticing. What is the register? Is the writing analytical, personal, prescriptive? How does each student relate to the product?

## Step 5 — Phase 1: Familiarise

Read all 13 documents without coding anything yet. In the document viewer, work through each respondent from top to bottom, reading every section. This will take 25–35 minutes.

As you read, keep a second memo open (create one titled `First Look`) and jot down:

- Moments that surprise you — claims, framings, or examples you did not anticipate
- Language the students use for their own experiences that you would not have chosen
- Silences: what is notably absent from the responses?
- Questions you cannot yet answer — moments you do not understand

Do not code yet. The purpose of this phase is to build a mental model of the corpus as a whole before you start labelling its parts.

{: .note }
> Notice as you read whether your codes-in-waiting respect the section boundaries. If you find yourself thinking "this is really about trust, not just informed consent," write that observation in your First Look memo. That is the data talking back to the schema.

After finishing all 13 documents, return to your Reflexivity memo and add a short paragraph: what were the three most surprising things you found? Where did the corpus depart most sharply from your expectations?

## Step 6 — Phase 2: Generate initial codes

Re-read the corpus systematically and begin coding. Select a segment of text that captures something analytically interesting, then apply a code.

**To create a code in Atlas.ti:**
- Select the text segment you want to code
- Right-click → **Code** → type a new code name, or select an existing one
- The segment becomes a *quotation* linked to that code

Codes should stay close to the data. A code like `caregiver has access to location data` is better than `privacy issue` — it describes what is actually happening in that segment, not just which predefined category it might belong to.

Keep coding through all 13 documents. Aim for 20–35 codes by the end. Some practical guidance:

**Code what is analytically interesting, not everything.** Not every sentence needs a code. Focus on segments that say something specific about the ethics of the wheelchair — a concrete scenario, a tension, a proposed design solution, an assumption that surprised you.

**Let codes multiply before merging them.** If two segments seem related but say something slightly different, give them different codes for now. You will merge and group in Phase 3.

**Watch for codes that span section boundaries.** If you apply the same code to a segment in the `=== Informed consent ===` section and a different segment in the `=== Trust ===` section, note this in your First Look memo. It is evidence that the predefined schema fragments something that belongs together.

**To see all your codes so far:** click the **Codes** tab in the left-hand panel to open the Code Manager. You can sort codes by frequency, rename them, or add a definition (right-click → Edit Comment).

## Step 7 — Phase 3: Construct candidate themes

Once you have coded the full sub-corpus, look across your codes for underlying patterns. A theme is not a topic label — it is an interpretive claim about meaning.

**Grouping codes into themes:**

1. In the Code Manager, read through all your codes. Look for codes that seem to share a central idea — not just the same subject, but the same interpretive claim.
2. Select codes that belong to the same theme → right-click → **Add to Code Group**. Name the group with a provisional theme title.
3. For each code group, write a **one-sentence theme definition** in a new memo titled `Candidate Themes`. The definition should be an argument, not a label:
   - `Caregiver access` → topic label
   - `The wheelchair creates a three-way accountability between user, caregiver, and manufacturer that no single consent process can resolve` → theme

Aim for 3–5 candidate themes. It is fine if some codes do not fit any theme yet — note them as outliers.

**The schema critique:**

In your `Candidate Themes` memo, add a short paragraph answering: which of your themes map cleanly onto the predefined columns, and which cut across two or more columns? For each theme that cuts across columns, name the columns it spans and explain why you think the schema fragments that concern.

This is not a criticism of the dataset — it is a finding about how the predefined structure shaped what students wrote, and how it might shape what you would find if you only analysed one column at a time.

{: .note }
> Atlas.ti's **Co-occurrence Table** (Analysis menu → Code Co-occurrence) shows which pairs of codes appear in the same documents most often. High co-occurrence between codes from different columns is quantitative evidence of thematic overlap — a useful check on the patterns you noticed while reading.

## Step 8 — Prepare for T5.3

Before closing Atlas.ti, record the following in your `Candidate Themes` memo:

1. **Theme inventory**: for each of your 3–5 themes, write: the theme name, the one-sentence definition, and the two or three codes that most strongly anchor it.

2. **Schema map**: for each theme, list which predefined columns it draws on. A theme that spans three columns is not a problem — it is a finding.

3. **Your prediction**: write one or two sentences completing this prompt — *"When I analyse the full Privacy column across all seven products in T5.3, I expect my wheelchair-derived theme [X] to..."* — and explain your reasoning. This prediction is what T5.3 will test.

Return to your Starboard notebook and paste this prediction into the expectations cell from Step 3. The two documents — your pre-reading expectations and your post-analysis prediction — will bracket what T4.3 taught you.

## Reflect

Before finishing, consider:

1. **The schema as a pre-analysis.** The six column headers are themselves an act of coding — someone decided that these six dimensions are the right way to organise ethics concerns. Did your inductive reading confirm that structure, or did it reveal concerns that the structure made hard to see? What would you have missed if you had only read one column?

2. **Positionality and the corpus.** You are a design student reading ethics assessments written by design students. Where did you find yourself nodding — and where did you feel the need to push back? Those moments of recognition and resistance are data about your analytical lens.

3. **What the text does not say.** Were there ethics concerns you expected to find that were largely absent from the wheelchair corpus? What might explain that absence — the product, the assignment framing, the student population, or something else?

4. **From exploration to participation.** The candidate themes you constructed are hypotheses, not findings. Which one would you most want to test by showing a wheelchair user — or a caregiver — and asking whether it matches their experience?
