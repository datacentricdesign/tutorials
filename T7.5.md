# T7.5 · LLM-Assisted Data Interface Generation

**Location:** `textbook/7/realisation.tex:274–277`
**Textbook context:** LLM-assisted prototyping; generating HTML/Streamlit from a description; iterating by conversation.
**Data Foundry features:** LocalAI, Existing Dataset, Website Hosting
**Platform pages:** [Local AI](https://data-foundry.net/Learning/LocalAI/index.html) · [Chatbots](https://data-foundry.net/Guides/LocalAI/Chatbots.html) · [JavaScript client](https://data-foundry.net/Guides/Clients/Javascript.html)

## Steps

1. **Prepare context** – Export a small sample of your dataset (10–20 rows of CSV) and write a concrete interface description: what data type, what user action, what fidelity level, what output format.
2. **Prompt an LLM** – Use Data Foundry's Local AI (or any code-capable LLM) with: description + sample data + desired output format (HTML, Streamlit, p5.js). Request a self-contained, immediately runnable file.
3. **Run & verify** – Open the generated file. Check: does it load data correctly? Are edge cases handled (empty arrays, missing values)? Is any data sent to an external server?
4. **Iterate by conversation** – Request 3 specific changes (layout, colour, filter control). Observe how precision of description affects output quality.
5. **Integrate with Data Foundry** – Replace the hardcoded CSV with a live fetch from an Existing or IoT Dataset using the JavaScript client. Host on Data Foundry.
6. **Reflect** – What kinds of interface requests did the LLM handle well vs. poorly? What still required manual coding? What are the privacy implications of the prompts you sent?
