# T3.2 · Designing a Data Donation Campaign

**Textbook context:** Data portability rights; data donation; designing collection studies; LLM-assisted prototyping.
{: .text-grey-dk-000 }

**Tools:** Agentic AI coding tool (tested with Claude Code), Data Foundry  
**Data Foundry features:** IoT Dataset, Existing Dataset, Website Hosting  
**Documentation pages:** [Data Protection](https://data-foundry.net/Learning/DataFoundry/DataProtection.html) · [JavaScript client](https://data-foundry.net/Guides/Clients/Javascript.html) · [Website Hosting](https://data-foundry.net/Guides/Hosting/WebsiteHosting.html)

## Introduction

In this tutorial you will design and build a working data donation campaign — from consent form to live hosted page — using an agentic AI coding tool like Claude Code to generate and iterate the implementation.

People who use connected services accumulate personal data that they legally own and can export under GDPR data portability rights. A **data donation** campaign asks them to voluntarily share that data for research or design purposes. Done well, it gives donors genuine agency: they understand what they are sharing, they see their own data before deciding, and they can withdraw at any time. Done poorly, it is just another data grab dressed up in friendly language.

{: .warning }
> **Ethics Approval**: *To conduct a data donation campaign you need an ethics approval from your institution.* Ask the teacher of your course how to proceed.

You will build a campaign for **Spotify streaming history** — a rich dataset covering listening patterns across time, device, and context. The end product is a self-contained single-page web app that:

- Explains the study and obtains explicit GDPR-compliant consent
- Lets donors load their Spotify export *locally in the browser* and actively explore their data — through both summary views and detailed drill-downs — before making any decision
- Lets them curate exactly what to donate: not only by selecting a time period, but by excluding individual entries or categories they are not comfortable sharing
- Uploads only the curated subset, together with a short survey, to Data Foundry

The emphasis is on **maximising understanding, not maximising donation**. A donor who has genuinely explored their data and made deliberate choices about what to share is more meaningfully informed than one who clicked through a summary screen. Beyond informed consent, the exploration step is itself an opportunity to offer donors something of value: a structured moment of self-reflection on how, when, and how much they use a service — insights they may never have had the occasion to see. The design challenge is to make that exploration feel genuinely useful, not like a hurdle before the real action.

Rather than writing the code yourself, you will **write prompts for an agentic AI coding tool** and iterate through conversation. This mirrors how LLM-assisted prototyping works in practice: your job is to be a clear, specific client — not a coder. This tutorial has been tested with [Claude Code](https://claude.ai/code); see the [alternatives section](#agentic-ai-coding-tool-alternatives) at the end if you are using a different tool.

By the end you will have:
- A working donation page hosted on Data Foundry
- A Data Foundry project that receives and stores survey responses and donated entries
- Experience turning a design brief into precise LLM prompts

{: .note }
> A key piece of advice from the Athlete case study (DCD textbook chapter 12): *"Be a pilot participant of your own study if you can! This is a good starting point to see what the data looks like and to empathise with what participants may experience."* Export and donate your own data first, before asking anyone else.

## Before you start

### Export your own Spotify data

Go through the export process yourself before designing instructions for anyone else.

1. Open Spotify → **Account** → **Privacy settings**
2. Scroll to **Download your data** → click *Request data*
3. Confirm via the email Spotify sends you
4. Wait for a second email with a download link (can take a few days — request it now and continue with the tutorial)
5. Unzip the archive. Look for files named `StreamingHistory_music_*.json`

Keep the unzipped folder to hand — you will use it to test your campaign.

{: .note }
> **No data yet?** Spotify exports can take up to a few days to arrive. If you have not received yours, use the synthetic dataset provided in [StreamingHistory_music_synthetic.json](https://raw.githubusercontent.com/datacentricdesign/tutorials/refs/heads/main/resources/T3.2/StreamingHistory_music_synthetic.json). It contains realistic listening patterns (no real person's data) and works with every step of this tutorial. You can replace it with your own export once it arrives.

### Set up your Data Foundry project

1. Go to [Data Foundry](https://datafoundry.io.tudelft.nl/) and create a project. Name it something that reflects your study.
2. In the **General** tab, add an **IoT Dataset** — this will store the intake survey responses. Name it `Survey`. Note its numeric **dataset ID** and **dataset token**.
3. In the **General** tab, add a second **IoT Dataset** — this will receive the donated streaming entries. Name it `Donations`. Note its **dataset ID** and **dataset token**.
4. If you plan to allow follow-up contact: add a third **IoT Dataset** named `Volunteers`. Note its **dataset ID** and **dataset token**. Keeping email addresses in a separate dataset, accessible only to the principal investigator, is a GDPR best practice. You could create this dataset in a separate Data Foundry project to better control access.
5. In the **General** tab, add an **Existing Dataset** named `www` — this will host your data donation web page publicly.

### Set up your agentic AI coding tool

This tutorial uses **Claude Code** as its agentic AI coding tool. The recommended setup is the **VS Code extension**, which gives you a chat panel alongside the editor — a more comfortable interface than the terminal for iterative prompting.

1. Install [Visual Studio Code](https://code.visualstudio.com/) if you do not have it already.
2. Open VS Code and go to the Extensions panel (`Ctrl+Shift+X` / `Cmd+Shift+X`). Search for **Claude Code** and install it.
3. Create a new folder for your project and open it in VS Code (`File → Open Folder`).
4. Click the Claude Code icon in the Activity Bar (left sidebar) to open the chat panel.

You will issue all prompts from this panel. Keep the session open throughout the tutorial — Claude Code uses the conversation history to understand each new request in context, and can read, create, and edit files directly in your project folder.

If you are not using Claude Code, see the [alternatives section](#agentic-ai-coding-tool-alternatives) at the end of this tutorial for equivalent setup instructions.

## How this tutorial works

Each step below has three parts:

1. **Brief** — the design or technical context you need to understand before writing a prompt
2. **Prompt questions** — questions to answer *before* writing your prompt; your answers become the prompt
3. **Check** — what to verify once the tool has generated the output

The goal is not to copy-paste a prompt. It is to practise translating a design intent into a precise, unambiguous instruction that an LLM can act on — and to notice where vagueness leads to poor output.

Your expertise as a designer is not replaced by the tool — it is the input the tool depends on. The ethical framing of the consent, the choice of what to ask in the survey, the decision to show donors their own data before asking them to share it: none of these come from the LLM. They come from you. The prompt questions in each step are designed to surface exactly that expertise, so that what you hand to the tool is a design decision already made — not a question left for it to answer.

## Step 1 · Scaffold the page

**Brief.** The first prompt sets the visual and structural skeleton: what kind of page, how many steps, what the UI should feel like, and what output format you need. A good scaffold prompt answers the "what" completely so that later prompts only need to answer the "what's in it".

The page will be a multi-step form. A progress indicator at the top helps donors understand where they are. The visual tone matters: a form that looks like a generic Bootstrap template reads differently from one that looks like it belongs to a university. You want donors to feel they are dealing with a credible institution, not a marketing funnel.

**Prompt questions.** Before writing your prompt, answer:

- What is the name of your study and the institution running it?
- How many steps does the form have, and what is each step called?
- What adjectives describe the visual tone you want (trustworthy? minimal? warm?)?
- What output format do you need — a single `index.html` you can open directly in a browser?
- What should the page *not* do yet (no submission logic, no real data processing — just navigation)?

Write your prompt and run it. Open the generated file in a browser.

**Check.**
- Can you click through all steps using the progress indicator?
- Does the visual tone feel appropriate for a university data collection study?
- What would a first-time visitor assume about who is running this and why?

## Step 2 · Add the consent section

**Brief.** Consent is not a formality — it is the ethical core of the campaign. Under GDPR Article 7, consent must be freely given, specific, informed, and unambiguous. For sensitive data (Article 9 categories — which include inferences about health, religion, and cultural identity that a listening history can support), explicit consent is required.

Three facts are easy to omit and important to include:

- The data will be used by *students*, not a professional research team. Donors deserve to know the actual context of use.
- Spotify data can reveal more than it appears to. A listening history spanning years may support inferences about mood states, religious practice, cultural identity, and daily routines. Donors should know this *before* they decide, not after.
- Withdrawal must be real. A withdrawal reference number — generated at donation time and held by the donor — is one practical mechanism. Without it, a "you can withdraw at any time" statement is hollow.

**Prompt questions.** Before writing your prompt, answer:

- What is the exact deletion date for the data? (GDPR requires a specific retention period, not "at the end of the study".)
- Who will have access to the data — student groups, course coordinator, anyone else?
- What is the email address for withdrawal requests?
- If you offer a follow-up conversation: who will contact the participant — the student group directly, or the coordinator first?
- What three things must donors explicitly confirm before continuing? (Think: age/location, informed agreement, acknowledgement of sensitive content.)
- Should the Continue button be disabled if any checkbox is unchecked? How should the error state look?

Write your prompt. Note that the consent text is long — give Claude Code all the specifics, including exact names, dates, and contact addresses. Vague prompts produce vague consent text that will not satisfy an ethics review.

**Check.**
- Try clicking Continue without checking all boxes. Does the validation work?
- Read the consent text aloud. Does it accurately describe *your* study?
- Is there anything a donor would reasonably want to know that is not covered?

## Step 3 · Add the file loader

**Brief.** A key principle of this campaign is that **no data leaves the donor's browser until they choose to share it**. The file loader parses the Spotify JSON locally, using the browser's `FileReader` API, and stores the entries in memory. Nothing is uploaded at this stage. This is not just a privacy measure — it is what makes the preview step possible.

Spotify has two export formats. The legacy format uses `endTime`, `artistName`, `trackName` and `msPlayed`. The extended format (available for accounts that have requested the full export) uses `ts`, `master_metadata_album_artist_name`, and `ms_played`. Your parser needs to handle both, since donors may have either version.

**Prompt questions.** Before writing your prompt, answer:

- What filename pattern should the upload zone accept? What file types?
- What instruction text should guide the donor through the Spotify export process? (Write it yourself, based on what you did in *Before you start*. Do not leave this to the tool — you have done the export and know the exact steps.)
- What feedback does the donor see after the file loads successfully? After a parse error?
- Should a "Preview" button appear only after parsing succeeds, or immediately on file selection?
- What should happen if a donor does not have their data yet — can they skip and return later?

Write your prompt.

**Check.**
- Load your own Spotify file. Does the Preview button activate?
- Try loading a non-Spotify JSON file. Does the error message make sense to a non-technical user?
- Open the browser developer tools (Network tab). Confirm that no request is sent when a file is loaded.

## Step 4 · Add the data exploration and curation

**Brief.** This step is the ethical heart of the campaign. Its goal is not to show donors just enough to feel informed — it is to give them the tools to *actually become* informed before they decide what to share. That requires two things working together: views that reveal what the data contains at different levels of detail, and controls that let donors act on what they see.

**Exploration** means the donor can move between scales. A summary view shows the shape of the whole dataset (how much data, over what period, with what patterns). A detail view lets them drill into any point of interest — a specific day, a specific artist, a specific time of week — and see the actual entries behind it. Without the detail view, the summary is abstract; without the summary, individual entries have no context. Both are necessary for genuine understanding.

**Curation** means the donor has meaningful control over what they share — not just a single date-range slider, but the ability to exclude specific entries or categories. A donor may be comfortable sharing their commute listening but not their late-night listening; comfortable sharing most artists but not one they consider private. Time-range filtering alone cannot express these distinctions. Per-entry or per-category exclusion gives donors the granularity to match their actual comfort level.

The Athlete case study (DCD textbook chapter 12) found that many donors had not previously explored their own data and were surprised by what it contained — and that *"at the time of donation, donors often don't know what they don't know"* about it. Your design task is to close that gap: not to minimise friction in the path to donation, but to maximise the quality of the consent that precedes it.

Closing that gap also creates an opportunity to offer donors something in return before they give anything away. A well-designed exploration step is itself a **personal data literacy experience**: donors may discover how much they listen, when, to what, and how that has shifted over time — insights they would not have sought out on their own. This reframes the value exchange. The campaign is not only asking donors for data; it is offering them a moment of structured self-reflection as part of the process. Designing for that — rather than treating the exploration step as a compliance checkbox — makes the campaign more honest about what it is doing and more genuinely useful to the people it asks to participate.

**Design decisions to make before writing your prompt.**

*Exploration — summary view:*
- What statistics should appear at a glance? (Plays, hours, date range, number of artists — what is most meaningful for this data type?)
- What visualisation best reveals listening patterns across time? A calendar heatmap (days as cells, colour = intensity)? A time-of-week scatter plot? Both?
- How many top artists or tracks should appear, and in what form?
- Should the data reveal anything that might surprise a donor — patterns they might not have expected? If so, how do you surface that without alarming or overwhelming?

*Exploration — detail view:*
- When a donor clicks on a day in the calendar, or an artist in the top list, what do they see? A list of individual plays with timestamps and durations?
- How far can they drill down? Can they inspect a single play event?
- What information is shown per entry — track name, artist, duration, time of day, device? All of these are in the export. What is relevant to a donor trying to understand what they are sharing?

*Curation:*
- Time-range filtering is the minimum. What finer-grained controls would give donors meaningful agency? Consider: excluding a specific artist entirely; excluding plays from a specific time of day (e.g. late-night); excluding plays below a minimum duration (which may represent accidental or skipped plays).
- When a donor excludes entries, how is that reflected in the summary? The curation summary line should always show the current state of the donation — not the original file.
- What happens if curation reduces the donation to zero entries? Should the donor be warned, or blocked, or simply informed?
- Should there be a "reset" option that restores the full data range and all excluded entries?

**Prompt questions.** Before writing your prompt, answer each design decision above. Then write a prompt that specifies:

1. The summary view: what statistics, what visualisations, what level of detail
2. The detail view: what clicking on a summary element reveals, down to individual entries
3. The curation controls: time range plus at least one additional per-entry or per-category exclusion mechanism
4. How the donation summary updates live as curation changes
5. Any warnings or friction points for edge cases (zero entries, very short date ranges)

This is the most complex step in the tutorial. Break it into sub-prompts if needed — for example, get the summary view working first, then add the detail drill-down, then add the curation controls.

**Check.**

This check is not only about verifying that the interface works — it is your first opportunity to experience the campaign as a participant would. Load your own Spotify data and go through the full exploration step as if you were deciding whether to donate. Notice what surprises you, what makes you uncomfortable, what you would want to exclude. That response is design feedback that no technical checklist can surface.

- Load your own Spotify file and navigate to the exploration view. Does the summary render correctly?
- Spend a few minutes genuinely exploring your data. What do you notice? Is there anything in your own history you would not want to share with a student group you have never met?
- Click a specific day or artist. Does the detail view open and show individual entries?
- Apply a time-range filter. Does the curation summary update immediately?
- Exclude a specific artist or category. Does the entry count in the summary decrease? Do excluded entries stay excluded if you adjust the date range?
- Reduce the donation to zero entries. What happens?
- Use the reset control. Does it restore the full dataset?
- Look at the detail view entries: could a donor reading them understand what each one represents — what they listened to, when, and for how long?
- After exploring, ask yourself: does the interface give you enough to make a genuinely informed choice? Is there anything you would still want to know or be able to see before committing? That gap is your next design iteration.

## Step 5 · Add the intake survey

**Brief.** The intake survey collects context that your data alone cannot provide: why people listen, on what devices, how actively they engage with music. Its placement in the flow — after the preview — is deliberate. A donor who has just seen their own listening history is in a different epistemic state than one who hasn't. The reflective question ("having just seen your data, what patterns do you recognise?") only makes sense here; placed before the preview it would ask about memory, not observation. This is iterative consent in practice: the donor's understanding deepens as they explore their data, and the study captures that reflection.

Survey design principles that apply here:
- Required questions should be the exception, not the rule. Only make a question required if you genuinely cannot use a response without it.
- Offer an "Other" option for any category list — you will not have anticipated every situation.
- The open text reflection question, placed directly after the preview, can surface insights no closed question would reach.

**Prompt questions.** Before writing your prompt, answer:

- What are your survey questions, in order? For each: what type of input (radio, checkboxes, textarea, email)?
- Which question, if any, is required before the user can continue?
- What is the exact wording of the reflection question?
- If a donor opts in for follow-up contact: should an email field appear inline? Should any validation apply?
- The "Donate" button ends this step. Should it submit both the survey answers and the filtered donation entries simultaneously, or in two separate actions?

Write your prompt. Include the full list of questions and their answer options verbatim — do not leave the tool to invent category labels.

**Check.**
- Walk through the full flow: Consent → Load file → Preview → Survey → Done. Does the progress indicator reflect this order?
- Does the validation on the required question work?
- Does the email field appear and disappear correctly based on the follow-up radio button?
- Read the reflection question in context. Does the wording feel natural after the preview?
- Read each question: is the wording neutral, or does it lead the donor toward a particular answer?

## Step 6 · Connect to Data Foundry

**Brief.** Data Foundry's IoT/Timeseries datasets accept anonymous token-based POST requests from a public page — no server-side code required. The endpoint pattern is:

```
POST /api/v1/datasets/ts/{datasetId}/{token}
Body: { device_id, activity, [param:value, param:value...] }
```

You need to post to three endpoints:

1. **Survey dataset** — one record per donor, containing all survey answers plus donation metadata (entry count, date range, withdrawal reference)
2. **Donations dataset** — the filtered streaming entries, as a CSV bulk upload via `/datasets/ts/logFile/{datasetId}`
3. **Volunteers dataset** — email address only, if the donor opted in, linked to the donor ID but not to the survey record

The third dataset is separate by design: it allows the principal investigator to facilitate follow-up contact without any other collaborator having access to donor email addresses.

Each donor should be assigned a stable anonymous ID for the session (`crypto.randomUUID()` stored in `sessionStorage`). This links the survey and donation records without identifying the donor across visits.

You will also generate a **withdrawal reference number** — a short human-readable code (e.g. `MML-4K9TXZ`) — that the donor keeps and can use to request deletion by email.

**Prompt questions.** Before writing your prompt, answer:

- What are your dataset IDs and tokens? (Have them ready — you will include them in the prompt, or in a `DF_CONFIG` object the tool generates.)
- What is your Data Foundry instance URL?
- What fields should the survey record contain?
- What columns should each CSV row in the donation bulk upload contain?
- What does the error message say if the upload fails?
- What does the thank-you screen show? What information does the donor need to keep?

Write your prompt, asking to create clear placeholder for your credentials and to tell you what to change/insert where. Test the submission live: load your own Spotify file, complete the flow, and check Data Foundry.

**Check.**
- After a successful submission, does the thank-you screen show the withdrawal reference number?
- Open Data Foundry: can you see the survey record? Can you see the donated entries?
- Open the browser developer tools (Network tab). Confirm no request is sent to any domain other than your Data Foundry instance.
- Try submitting with the network offline. Does the error handler catch it and re-enable the button?

## Step 7 · Deploy to Data Foundry

**Brief.** An Existing Dataset named `www` with web access enabled serves `index.html` at your project's public URL. This gives you a stable, hosted link you can distribute to participants without any external hosting dependency.

**Prompt questions.** Before writing your prompt, answer:

- Are there any placeholder values still in the HTML (e.g. `[coordinator-email]`, demo mode flags)? List them — your prompt should ask Claude Code to confirm they are all filled in.
- What should the page title and browser tab title say?
- Is there anything you noticed during testing that needs a final fix before deployment?

Write your prompt asking for a production-ready `index.html` and the exact upload steps.

After uploading, distribute the URL to one classmate and observe what they experience as a first-time donor. Note anything that confuses them or that they read differently than you expected.

**Check.**
- Open the page in a private/incognito window. Walk through the full flow as a new donor.
- Submit a test donation. Verify it arrives in Data Foundry under a different donor ID than your previous tests.
- Check the browser console for any errors.

## Step 8 · Monitor incoming donations (optional)

**Brief.** Once donations start arriving, you need a way to see what you have. A short Starboard notebook snippet can pull the survey records from Data Foundry and produce a quick overview: how many donors, what age distribution, what music uses. Following a similar approach as each step of these tutorial (e.g., brief, questions, prompt, check), you could quickly develop a web page that shows statistics about your data donation.

## Reflect

Before using this data in a design project, consider:

1. **What metadata reveals.** The Spotify dataset holds play counts, timestamps, and artist names — not song lyrics, not comments, not explicit content. Yet from it you can infer when someone wakes up, whether they listen during religious observance, what emotional state they were in. Does "no sensitive content" mean "not sensitive"? How does your answer change your obligations as a collector?

2. **The platform shapes the data.** Donors who requested the full export get a different format (and more fields) than those who got the legacy export. Some donors may have used Spotify for years; others for months. Some may have family members on the same account. How does this affect what you can conclude from a cross-donor analysis?

3. **Preview as iterative consent.** The Athlete case study found that donors *"at the time of donation, often don't know what they don't know"* about their own data. The preview step is designed to address this. Did any of your pilot donors report noticing something surprising? Did anyone choose to narrow their date range after seeing the preview? What does that tell you about the difference between informed and uninformed consent?

4. **Prompting as design.** Each step of this tutorial asked you to answer questions before writing a prompt. Where did the tool produce better output when you were more specific? Where did it go wrong when you left something vague? What does this suggest about who bears responsibility for the quality of LLM-generated code in a real project?

5. **Value exchange.** What do participants gain by donating their Spotify data? How does this compare to the value the study receives? Would you feel comfortable making the value exchange explicit in the consent form — and if not, why not?

## Agentic AI coding tool alternatives

An **agentic AI coding tool** is an LLM that can read and write files, run terminal commands, and iterate on code across multiple steps — rather than just answering questions in a chat window. Claude Code is one option; several alternatives are available, some free.

| Tool | Model | Free tier | How to start |
|------|-------|-----------|--------------|
| **[Claude Code](https://claude.ai/code)** | Claude (Anthropic) | No — requires a Claude Pro or API subscription | `npm install -g @anthropic-ai/claude-code` then `claude` in a terminal |
| **[GitHub Copilot](https://github.com/features/copilot) in VS Code** | GPT-4o / Claude (configurable) | Free for students and open-source contributors via [GitHub Education](https://education.github.com/pack) | Install the *GitHub Copilot* and *GitHub Copilot Chat* extensions in VS Code; use **Agent mode** in the chat panel |
| **[Cursor](https://www.cursor.com/)** | GPT-4o / Claude (configurable) | Free tier (limited monthly uses) | Download and install Cursor (a VS Code fork); the agent is available in the chat sidebar |
| **[Windsurf](https://windsurf.com/)** | GPT-4o / Claude / own models | Free tier | Download and install Windsurf; use the *Cascade* panel for agentic tasks |
| **[Continue](https://www.continue.dev/) + local model** | Any model via Ollama | Free — runs entirely on your machine | Install the *Continue* VS Code extension; configure it to use a local model via [Ollama](https://ollama.com/) |

### What to look for in any tool

Whatever tool you use, it should be able to:

- Read and write files in your working directory
- Hold a **persistent conversation** across multiple prompts (so later prompts build on earlier ones)
- Run in or alongside a code editor so you can inspect and test the generated output immediately

A plain chat interface (Claude.ai, ChatGPT web) can generate the code, but you will need to copy-paste each result manually — which breaks the iterative flow this tutorial relies on.
