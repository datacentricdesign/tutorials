# T7.1 · Setting Up a Sensor Probe with the Connected Interaction Kit

## Introduction

In this tutorial you will build a wireless sensor probe using the **Connected Interaction Kit** — a prototyping platform developed at the faculty that is designed for sensing, actuating, and communicating over WiFi. The kit's **BitsyExpander** board carries an ESP32 co-processor that handles all networking, so your main program can focus entirely on reading sensors and sending data.

Data is transmitted over **OOCSI** — an open message-passing protocol maintained at TU/e — to the Data Foundry platform, where it is stored in an IoT Dataset that you can inspect in real time and download for analysis. OOCSI function like MQTT as a publish/subscribe model.

By the end of this tutorial you will have:

- A physical probe that reads distance every second and sends each reading to Data Foundry.
- A live IoT Dataset in your project containing a time-series of sensor values.
- A working understanding of how hardware, firmware, and a data platform connect end-to-end.

The example uses a **VL53L0X Time-of-Flight (ToF) distance sensor**, but the same firmware structure applies to any I²C sensor supported by CircuitPython.

---

## Before you start

### Hardware you need

- Connected Interaction Kit with a **BitsyExpander** (includes the ESP32 WiFi co-processor)
- **VL53L0X** ToF distance sensor (included in the kit)
- USB cable for power and programming
- Access to a WiFi network (on the TU Delft campus: `TUD-facility`)

### Set up your Data Foundry project

1. Go to [Data Foundry](https://datafoundry.io.tudelft.nl/) and open (or create) a project.
2. In the **General** tab, add an **IoT Dataset**. Name it `Sensor Probe` and note its numeric **dataset ID** shown in the top-left corner of the dataset page.
3. In the **Sources** tab, add a device. Only the name is required — call it something like `probe-01`. Once created, copy its **device ID** (a long hexadecimal string).
4. On the IoT Dataset page, scroll down to **Configuration** and select the **Data from OOCSI** tab. Type in the OOCSI channel name you will use for this probe (this is the `oocsi_topic` you will set in `settings.py` in Step 2), then click **Save**.

{: .note }
> The device ID in Data Foundry is *not* the same as the WiFi MAC address. It is a unique identifier assigned by the platform and is what links incoming OOCSI messages to your specific probe.

## Step 1 — Wire up the hardware

The VL53L0X sensor has a **Grove connector**. Plug it into one of the Grove ports on the BitsyExpander labelled **I2C** — no wiring or pin mapping needed.

The BitsyExpander exposes the I²C bus on those Grove ports as `board.SDA` / `board.SCL`. No additional pull-up resistors are required — the board provides them.

{: .note }
> The VL53L0X measures distances from roughly 30 mm to 1200 mm with millimetre resolution. It works best on diffuse, non-glossy surfaces. Transparent or mirror-like surfaces may give erratic readings.

---

## Step 2 — Configure the firmware

The full firmware lives in the [resource folder of tutorial T7.1](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.1). It consists of two files you need to edit:

### `settings.py` — your credentials

Open [settings.py](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.1/settings.py) and fill in your values:

```python
settings = {
    "ssid":                 "TUD-facility",       # WiFi network name
    "password":             "your-wifi-password", # WiFi password (registered for your device on campus)
    "device_id":            "your-device-id",     # Device ID from Data Foundry (Sources tab)
    "oocsi_base":           "oocsi.id.tue.nl",    # OOCSI server — do not change
    "oocsi_topic":          "/study/living-room", # OOCSI topic — choose your own (see below)
    "activity":             "MEASURING",          # Activity label stored with each data point
    "collection_frequency": 1                     # Seconds between readings
}
```

**Choosing an `oocsi_topic`.**  The topic is a free-form string that acts as an address on the OOCSI network. Slashes (`/`) create a hierarchy that makes topics easier to organise when you have multiple probes: `/study/kitchen`, `/study/bedroom`, `/prototype/v2`, etc. Pick something meaningful for your project — Data Foundry listens on the topic you configure for your IoT Dataset and stores every message that arrives on it.

`device_id` and `activity` are sent inside every OOCSI message so that Data Foundry can attribute readings to the right source and tag them with a meaningful label. The `activity` field is useful when the same probe goes through distinct phases (e.g. `BASELINE`, `INTERVENTION`, `WASHOUT`) — you change it in `settings.py` before each phase, redeploy, and the dataset records which phase each reading belongs to.

### `code.py` — the main program

You do not need to modify [code.py](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.1/code.py) to run the example. Reading through it is worthwhile to understand the structure:

```python
# 1. Connect to WiFi via the BitsyExpander's ESP32 co-processor
esp = adafruit_esp32spi.ESP_SPIcontrol(spi, esp32_cs, esp32_ready, esp32_reset)
# ...scan and connect...

# 2. Open an OOCSI connection to the Data Foundry relay
oocsi = OOCSI(settings["device_id"], settings["oocsi_base"], esp)

# 3. Initialise the ToF sensor on the I²C bus
dist_sensor = adafruit_vl53l0x.VL53L0X(i2c_port)

# 4. Read and send in a loop
while True:
    distance = dist_sensor.range
    oocsi.send(settings["oocsi_topic"], {
        "device_id": settings["device_id"],
        "activity":  settings["activity"],
        "distance":  distance
    })
    time.sleep(settings["collection_frequency"])
```

Each call to `oocsi.send()` delivers a JSON message to the OOCSI server, which routes it to the Data Foundry IoT Dataset associated with your device ID. The message arrives with a server-side timestamp, so you do not need to implement clock synchronisation on the device.

## Step 3 — Deploy to the board

1. Connect the BitsyExpander to your computer via USB. It mounts as a small drive called `CIRCUITPY`.
2. Copy the entire [resource folder of tutorial T7.1](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.1) folder contents onto the `CIRCUITPY` drive, preserving the `lib/` subdirectory.
3. CircuitPython will automatically runs `code.py` on boot.
4. Open a serial monitor (e.g. the one built into the [Mu editor](https://codewith.mu/), or `screen /dev/tty.usbmodem* 115200` on macOS) to watch the startup messages.

On a successful boot you should see something like:

```
ESP32 WiFi Module found.
Firmware version: 1.7.4

Scanning for available networks...
TUD-facility found. Connecting...
Connected! IP address: 192.168.1.42
*** OOCSI: [your-device-id]: connecting to oocsi.id.tue.nl port 4444
*** OOCSI: [your-device-id]: Connection established
Sending distance: 312
Sending distance: 308
Sending distance: 315
```

If you see `not found` for the WiFi network, check that `ssid` in `settings.py` matches exactly (case-sensitive). If the OOCSI connection fails, verify that `oocsi_base` is `oocsi.id.tue.nl` and that the device is on a network with outbound TCP access on port 4444.

---

## Step 4 — Verify data in Data Foundry

1. Open your project in Data Foundry and navigate to the **Sensor Probe** IoT Dataset.
2. Click **Data Table**. New rows should appear every second.
3. Move your hand toward and away from the sensor and watch the `distance` column change (refresh the page to see new entries).
4. Confirm that the `device_id` and `activity` columns match what you set in `settings.py`.

{: .note }
> The first row may take 10–15 seconds to appear after the board connects, as OOCSI requires a brief handshake before messages are forwarded to Data Foundry.

## Step 5 — Adjust sampling rate and labelling

The `collection_frequency` setting is the simplest knob for managing data volume and battery life:

| Frequency | Use case |
|-----------|----------|
| 1 s | Rapid prototyping, short sessions |
| 10 s | Continuous deployment over hours |
| 60 s | Multi-day ambient monitoring |

A reading every second produces 86 400 rows per day per device. That is manageable for a short study, but if you are running several probes simultaneously, or deploying for a week, a slower rate keeps the dataset tractable.

For a **7-day deployment** at 1-second resolution: 7 × 86 400 = 604 800 rows. Consider whether that temporal resolution is necessary for your research question before committing to it.

Try changing `collection_frequency` to `10` in `settings.py`, saving the file, and confirming the row arrival rate in Data Foundry drops accordingly. No other change is needed.

## Reflect

Before deploying this probe in a study context, consider:

1. **Sensor choice and validity.** A ToF sensor measures distance — but what phenomenon in your study does distance stand in for? Is that a valid proxy, and under what conditions does the mapping break down?

2. **Sampling rate and data quality.** The current code takes one reading per second and immediately sends it — sampling rate and transmission rate are the same. A different approach is to sample more frequently (e.g. ten times per second), accumulate the values, and send only the average once per second. This reduces noise without increasing data volume. Faster is not always better as a raw feed either: look at your data and ask whether single-point readings are stable enough for your purpose, or whether on-device averaging would give you a more meaningful signal to store.

3. **Participant awareness.** The probe is a physical object in someone's space. What does it communicate to them about what is being measured? Does the device's presence change the behaviour you are trying to observe?

4. **Data coverage.** WiFi dropout, power interruptions, and sensor range errors will leave gaps in your time-series. Before fielding, decide: how will you detect gaps, and what will you do with them in analysis?
