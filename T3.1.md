## Introduction

In this tutorial you will use the **NS API** (Dutch Railways) to collect real-world transit data programmatically. It is a good first API: free for students, well-documented, and returns data about places and events that are easy to reason about.

You will retrieve two types of data:

- **Stations** — NS stations nearest to a given location
- **Disruptions** — current service disruptions, per station or nationwide

By the end you will have a dataset stored in Data Foundry and a basic profile of its contents.

---

## Before you start

### Get an NS API key

1. Go to [apiportal.ns.nl](https://apiportal.ns.nl). Create a free account by clicking 'Inloggen', then 'Externe bezoeker'. It will ask for your email address and send you a verification code.
2. Once logged in, click 'API Products' and search for 'NS-App'. Select the result, enter a subscription name (e.g. 'My NS App subscription'), and click 'Subscribe'.
3. Copy your **Primary Key** from the product page. You can retrieve it at any time via the 'Profiel' page.

{: .note }
> Keep your API key out of shared notebooks. Do not commit it to version control. In this tutorial, paste it only in your own private notebook session.

### Set up your Data Foundry project

1. Go to [Data Foundry](https://datafoundry.io.tudelft.nl/) and create a project (or open an existing one).
2. In the tab 'General', Add an **Existing Dataset** — this will host the Starboard Notebook. Name it `NS Disruptions`.
3. In the tab 'General', add an **IoT Dataset** — this will store the disruption records you collect. Name it `NS Disruptions IoT` and note its numeric **dataset ID** from the top left corner.
4. In the tab 'Sources', add a device. Only the name is required, you could call it 'ns-api'. Once created, take note of its ID.
5. Go to [Data tools > Notebooks](https://datafoundry.io.tudelft.nl/tools/starboard) and click 'Add a New Notebook'. Choose your project and the Existing Dataset you just created, and give it a name (e.g. 'My NS Notebook').

## Step 1 — Query the NS API

Open the newly created notebook. It comes with default JavaScript and Python cells. You can delete the JavaScript cell by clicking the three vertical dots in its top right corner.

{: .note }
> All cells in Starboard run in an async context, so `await` works at the top level.

### Configure your credentials

```python
# Your NS API key — paste the Primary Key you copied from the NS API portal
NS_API_KEY = "paste-your-ns-api-key-here"

# The base URL of the NS API — all requests start with this address
NS_BASE    = "https://gateway.apiportal.ns.nl"

# The NS API uses a header to verify your identity on every request.
# Think of it as showing your badge before entering a building.
NS_HEADERS = {"Ocp-Apim-Subscription-Key": NS_API_KEY}
```

### Find stations near a location

The endpoint returns NS stations closest to a latitude/longitude pair. Use the coordinates of your own location, or TU Delft (52.0024, 4.3704) to start.

{: .note }
> If you are familiar with fetching API data in Python, you most likely use `requests`. Because Starboard Notebooks run Python in the browser via [Pyodide](https://pyodide.org), synchronous networking is not available — we use `pyfetch` instead, which works the same way but is asynchronous.

```python
from pyodide.http import pyfetch  # the tool we use to make HTTP requests in Starboard

# Your location as latitude and longitude — change these to your own coordinates
lat, lng = 52.0024, 4.3704   # TU Delft

# Send a request to the NS API asking for the 5 nearest stations to our location.
# The f-string builds the full URL by inserting our variables into the address.
resp = await pyfetch(
    f"{NS_BASE}/nsapp-stations/v2/nearest?lat={lat}&lng={lng}&limit=5",
    headers=NS_HEADERS
)

# Convert the response from raw text into a Python dictionary, then extract
# the "payload" key which contains the list of stations
stations = (await resp.json())["payload"]

# Loop through each station and print its full name, code, and coordinates
for s in stations:
    print(s["namen"]["lang"], "—", s["code"], f"({s['lat']}, {s['lng']})")
```

For the TI Delft location, the result looks as follows:

```
Delft — DT (52.0066680908203, 4.35638904571533) 
Delft Campus — DTCP (51.9908332824707, 4.36472225189209) 
Rijswijk — RSW (52.039722442627, 4.31916666030884) 
Den Haag Ypenburg — YPB (52.0552787780762, 4.39222240447998) 
Den Haag Moerwijk — GVMW (52.0538902282715, 4.3086109161377) 
```

Note the **station code** of your nearest station (e.g. `DT` for Delft). You will use it below.

{: .note }
> Starboard Notebooks lives in your browser, nothing is saved by default. Regularly go to the top of the notebook and click 'Save notebook' to save a copy on Data Foundry.

### Get disruptions at that station

```python
# Replace "DT" with the station code printed by the previous cell
station_code = "DT"

# Request all disruptions currently affecting this station
resp = await pyfetch(
    f"{NS_BASE}/disruptions/v3/station/{station_code}",
    headers=NS_HEADERS
)
# Convert the response to a Python list of disruption records
station_disruptions = await resp.json()

# Print a summary: total count, then the first 5 disruptions
print(f"{len(station_disruptions)} disruption(s) for {station_code}:")
for d in station_disruptions[:5]:
    # .get() safely reads a field — it returns None instead of an error if the field is missing
    print(" -", d.get("type"), "|", d.get("title"))
```

If everything is smooth at your nearest station, you should see something like:

```
0 disruption(s) for DT: 
```

### Get all active disruptions nationwide

```python
# Request all disruptions currently active across the entire NS network.
# "?isActive=true" is a query parameter — it tells the API to filter results.
resp = await pyfetch(
    f"{NS_BASE}/disruptions/v3?isActive=true",
    headers=NS_HEADERS
)
# Store the full list in a variable — we will use it in the next steps
all_disruptions = await resp.json()

print(f"Active disruptions nationwide: {len(all_disruptions)}")
```

This code should produce an output that looks like:

```
Active disruptions nationwide: 2 
```

## Step 3 — Flatten to a table

Convert the disruptions list to a pandas DataFrame so it is easy to inspect and upload.

```python
import pandas as pd                      # the standard Python library for working with tables
from datetime import datetime, timezone  # used to record when we collected the data

# Build a flat list of rows — one row per disruption, keeping only the fields we care about
rows = []
for d in all_disruptions:
    # Disruptions can have multiple timespans (e.g. repeated maintenance windows).
    # We extract cause and situation from the first one, which covers most cases.
    first_timespan = d.get("timespans", [{}])[0]

    rows.append({
        "id":           d.get("id"),
        "type":         d.get("type"),       # e.g. "DISRUPTION" or "MAINTENANCE"
        "title":        d.get("title"),
        "is_active":    d.get("isActive"),
        "start":        d.get("start"),      # disruption start time (ISO 8601 string)
        "end":          d.get("end"),        # may be None if the disruption is still ongoing
        "impact":       d.get("impact", {}).get("value"),  # severity score (integer)
        # cause and situation are nested inside the first timespan
        "cause":        first_timespan.get("cause", {}).get("label") if first_timespan.get("cause") else None,
        "situation":    first_timespan.get("situation", {}).get("label") if first_timespan.get("situation") else None,
        # Record exactly when we ran this collection — useful if you repeat it later
        "collected_at": datetime.now(timezone.utc).isoformat()
    })

# Convert the list of rows into a DataFrame (a table with named columns)
df = pd.DataFrame(rows)

# df.shape returns (number of rows, number of columns)
print(df.shape)

# df.head() shows the first 5 rows as a formatted table
df.head()
```

## Step 4 — Store in Data Foundry

So far, all the data only lives inside the notebook. In this step you will push each disruption to your **NS Disruptions IoT** dataset in Data Foundry, one record at a time.

This follows the same pattern used by physical sensors: a device sends a JSON payload to a URL, and the platform stores it with a timestamp. The only difference is that here the "device" is your notebook.

You need the **dataset token** for your IoT dataset — find it on the dataset page in Data Foundry (not your account token). You also need the ID of your device 'ns-api' created at the start of this tutorial.

{: .note }
> The dataset token is a base64 string that often ends with `=`. Because the token is embedded in the URL path, that `=` must be replaced with `%3D` — otherwise the server cannot match the route and returns a 404. The line `DATASET_TOKEN.replace("=", "%3D")` handles this automatically.

```python
import json  # used to convert Python dictionaries to JSON text

DF_BASE       = "https://datafoundry.io.tudelft.nl"
SOURCE_ID     = "paste-your-source-reference-id" # The ID of your ns-api device created at the start of this tutorial
DATASET_TOKEN = "paste-your-dataset-token-here"  # find this on the IoT dataset page
DATASET_ID    = 123  # replace with the dataset ID of your NS Disruptions IoT dataset

# POST each disruption as a separate record to the IoT dataset
for d in all_disruptions:
    first_timespan = d.get("timespans", [{}])[0]

    # Each record has three fields:
    # - source_id: identifies where the data came from (here, the disruption ID)
    # - activity:  a label describing what kind of event this is
    # - data:      the payload — a JSON string containing the fields we care about
    record = {
        "source_id": SOURCE_ID,        # identifies the data source — fixed label, not per-disruption
        "activity":  d.get("type"),   # e.g. "DISRUPTION" or "MAINTENANCE"
        "data": json.dumps({          # json.dumps() converts a dict to a JSON string
            "title":     d.get("title"),
            "is_active": d.get("isActive"),
            "start":     d.get("start"),
            "end":       d.get("end"),
            "impact":    d.get("impact", {}).get("value"),
            "cause":     first_timespan.get("cause", {}).get("label") if first_timespan.get("cause") else None,
            "situation": first_timespan.get("situation", {}).get("label") if first_timespan.get("situation") else None,
        })
    }

    # Send the record to Data Foundry with a POST request.
    # Unlike the GET requests above, POST sends data — the record is in the request body.
    resp = await pyfetch(
        f"{DF_BASE}/api/v1/datasets/ts/{DATASET_ID}/{DATASET_TOKEN}",
        method="POST",
        headers={"Content-Type": "application/json"},
        body=json.dumps(record)  # convert the record dict to a JSON string for transmission
    )
    print(f"Posted {d.get('id')} — status {resp.status}")
```

When running the code, you should get something the looks like the following. Note the status (200), which means that the HTTP request successfully posted the data on Data Foundry. If you get a different code, there is an error somewhere.

```
Posted Engineering works Hoorn-Enkhuizen, 11 to 22 April — status 200 
Posted Engineering works between Sittard and Maastricht 11 to 16 April — status 200 
Posted Engineering works between Schiphol Airport - Amsterdam RAI 11 and 12 April — status 200 
Posted Groningen - Leer. — status 200 
Posted Amsterdam - Utrecht. — status 200 
Posted Amsterdam - Köln - Zürich HB. — status 200 
```

You can now go to your Data Foundry project, open the NS IoT Dataset and click 'Data Table'. You should this a table with the NS data you just sent.

## Step 5 — Profile the dataset

With the data stored, run a quick profile.

```python
# How many rows and columns does the table have?
print("Shape:", df.shape)

# What type of data is stored in each column? (e.g. text, number, boolean)
print("\nColumn types:")
print(df.dtypes)

# How many values are missing (None / NaN) per column?
# Missing values are common in real-world APIs — a disruption might not have an end time yet.
print("\nMissing values:")
print(df.isnull().sum())

# How many disruptions of each type are there?
print("\nDisruption types:")
print(df["type"].value_counts())

# What are the most common causes?
print("\nTop causes:")
print(df["cause"].value_counts().head(10))

# What impact scores appear? Higher values indicate more severe disruptions.
print("\nImpact scores:")
print(df["impact"].value_counts().sort_index())
```

## Reflect

Before using this data in a design project, consider:

1. **Collection context.** NS publishes disruption data for passenger information systems. Does that original purpose align with using it as design research material?

2. **What is absent.** The disruptions feed describes service failures — not how many people were affected, how long they waited, or how they coped. What additional data would you need to understand the *human experience* of a disruption?

3. **Aggregation and identifiability.** Disruption data describes routes and stations, not individuals. But if you combined it with check-in/check-out data (OV-chipkaart), how would the privacy picture change?

4. **Snapshot vs. time series.** You captured one moment in time. How would your analysis change if you collected this data hourly for a week? What design questions would become answerable?
