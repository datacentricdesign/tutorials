# T7.3 · End-to-End Stack: Sensor → Data Foundry → Actuator

## Introduction

This tutorial walks through a complete data pipeline from a body-worn sensor to a physical actuator, with Data Foundry doing the processing and storage in between. The scenario is a **sedentary behaviour reminder**: eSense earbuds stream head-movement data via Bluetooth to a browser page, which computes a motion metric and forwards it to Data Foundry over OOCSI. A script running on Data Foundry detects when the participant has been still for too long and sends a trigger event to a Connected Interaction Kit prototype, which responds with a short vibration pattern.

```
eSense earbuds (IMU)
  → Bluetooth → Browser page
    → OOCSI → Data Foundry IoT Dataset
      → Data Foundry Script (activity classifier)
        → OOCSI → Connected Interaction Kit
          → vibration motor
```

**Why does the classification run on Data Foundry and not in the browser page?** In a finished product you would almost certainly move this logic into the app — closer to the sensor, with no dependency on a server, and able to run offline. During prototyping the balance shifts: a script on Data Foundry can be edited, saved, and redeployed in seconds without touching the participant's device or the firmware. Keeping the classifier server-side is a deliberate prototyping strategy, not an architectural recommendation. Once the threshold values and timing are validated in the wild, the same logic can be ported into the app with confidence.

By the end of this tutorial you will have:

- A browser page that receives live IMU data from eSense earbuds via Web Bluetooth, packages it into 2-second windows, and forwards each window to Data Foundry.
- An IoT Dataset in your project containing a time-series of activity scores.
- A Data Foundry Script that classifies activity and sends a trigger event after prolonged stillness.
- A Connected Interaction Kit prototype that receives the trigger and vibrates.
- Experience with remote iteration: changing the stillness threshold on the server without touching the participant's setup.

## Before you start

### Hardware you need

- A pair of eSense earbuds (left earbud only is required)
- A Connected Interaction Kit with a BitsyExpander
- A vibration motor connected to the Grove connector on pin **D4**
- A USB cable for power and programming
- A computer running **Chrome** (Web Bluetooth is a Chrome feature — Firefox and Safari do not support it)
- Access to a WiFi network (on campus: `TUD-facility`)

### Set up your Data Foundry project

You will need **two OOCSI channels**: one that the browser page publishes to (the sensor channel), and one that the Data Foundry Script publishes to (the actuator channel). Choose names that are unique to your project, for example `study/esense-data` and `study/actuator`.

1. Open (or create) a project in [Data Foundry](https://datafoundry.io.tudelft.nl/).
2. In the **General** tab, add an **IoT Dataset**. Name it `eSense Activity` and note its numeric **dataset ID**.
3. On the dataset page, go to **Configuration → Data from OOCSI** and enter your OOCSI channel name (e.g. `study/esense-data`). Click **Save**. Data Foundry will now store every OOCSI message that arrives on that channel.
4. In the **Sources** tab, add a device. Call it something like `esense-actuator`. Copy its **device ID**.

{: .note }
> The device ID in Data Foundry is a long hexadecimal string assigned by the platform. It is distinct from the device name you choose and is what OOCSI uses to route messages correctly.

## Step 1 — Stream IMU data from the eSense earbuds

The [resource folder of tutorial T7.3](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3) contains `esense-imu.html`, a self-contained browser page that connects to the left eSense earbud via Web Bluetooth, computes a windowed activity score from the accelerometer, and forwards each window to Data Foundry over OOCSI.

### Open the page in Chrome

Download both files from the resource folder and place them in the **same directory**:

- [esense-imu.html](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3/esense-imu.html) — the sensor page
- [oocsi-web.min.js](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3/oocsi-web.min.js) — the OOCSI WebSocket library loaded by the page

Then open `esense-imu.html` directly in Chrome — no local server needed. Chrome allows Web Bluetooth from local files.

{: .note }
> **Why Chrome?** Web Bluetooth is a Chrome initiative. Firefox has explicitly decided not to implement it; Safari has partial support on macOS but not for the BLE service UUIDs used by eSense. For anything you build on top of this tutorial, Chrome (desktop) is the only browser you can rely on.

### Configure OOCSI

In the **OOCSI / Data Foundry** panel on the page:

| Field | Value |
|---|---|
| Channel | Your sensor channel (e.g. `study/esense-data`) — must match what you entered in the Data Foundry dataset configuration |
| Client name | Any unique string, e.g. `esense_client_01` |
| Activity label | A short label for the current study phase, e.g. `TEST` |
| Window | `2` (seconds) |

Click **Connect OOCSI**. The status badge should turn yellow and read "connected".

### Connect the earbud and start streaming

1. Make sure the left eSense earbud is switched on and not connected to any other device.
2. Click **Connect eSense**. A browser picker will appear — select the earbud (its name is `eSense-XXXX`).
3. Once connected, click **Start sampling**. The page starts sending one OOCSI message every 2 seconds.

Each message contains:

| Field | Description |
|---|---|
| `device_id` | Bluetooth device name of the earbud, e.g. `eSense-1234` |
| `activity` | Study phase label set in the OOCSI panel |
| `activity_score` | Std dev of accelerometer magnitude over the window (g) — near 0 when still, rises with movement |
| `acc_x`, `acc_y`, `acc_z` | Mean acceleration per axis (g) |
| `gyro_x`, `gyro_y`, `gyro_z` | Mean rotation rate per axis (deg/s) |
| `samples` | Number of IMU packets received in this window |

### Verify data arrives in Data Foundry

Open your `eSense Activity` dataset in Data Foundry and click **Data Table**. New rows should appear every 2 seconds. Check that `activity_score` is close to zero when you sit still, and rises when you move your head.

{: .note }
> **Calibrating the threshold.** Sit still for about 30 seconds and note the `activity_score` values in the table. The stillness threshold in the Data Foundry Script (Step 3) should be set just above this resting noise floor. A value around `0.02` is a typical starting point.

### Host the page on Data Foundry

For local testing, opening the file directly in Chrome is sufficient. When you are ready to deploy to a participant, you can host the page on Data Foundry so that it is accessible from any device without file transfers.

1. In your Data Foundry project, go to the **General** tab and add an **Existing Dataset**.
2. Upload both `esense-imu.html` and `oocsi-web.min.js` to the dataset.
3. Open the dataset's **Configuration** tab and select **Web Access**.
4. Click **Activate web access**. Data Foundry will provide a public URL.

Share that URL with the participant — they open it in Chrome, connect the earbud, and start streaming without installing anything.

## Step 2 — Set up the actuator firmware

Open [settings.py](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3/settings.py) from the resource folder and fill in your values:

```python
settings = {
    "ssid":                  "TUD-facility",
    "password":              "your-wifi-password",
    "device_id":             "your-device-id",       # from Data Foundry Sources tab
    "oocsi_actuator_channel": "your-esp32-channel",  # channel the script will publish to
    "oocsi_base":            "oocsi.id.tue.nl",
}
```

`oocsi_actuator_channel` is the channel the Data Foundry Script will publish trigger events to. Choose a name and keep it — you will need to enter the same name in the script in the next step.

### Deploy to the board

1. Connect the BitsyExpander to your computer via USB. It mounts as a drive called `CIRCUITPY`.
2. Copy the contents of the [esp32 folder](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3/esp32) onto the `CIRCUITPY` drive, preserving the `lib/` subdirectory.
3. CircuitPython automatically runs `code.py` on boot.
4. Open a serial monitor to watch the startup messages.

On a successful boot you should see:

```
ESP32 WiFi module found.
Firmware version: 1.7.4

Scanning for available networks...
TUD-facility found. Connecting...
Connected! IP address: 192.168.1.55

Connecting to OOCSI...
*** OOCSI: [your-device-id-XXXX]: connecting to oocsi.id.tue.nl port 4444
*** OOCSI: [your-device-id-XXXX]: Connection established
*** OOCSI: [your-device-id-XXXX]: subscribed to your-esp32-channel
Ready. Waiting for sedentary alerts...
```

The built-in LED on D13 will blink once per second as a heartbeat. The vibration motor on D4 will pulse three times shortly — you can test this by sending a message manually from an OOCSI client.

### Vibration pattern

The firmware triggers `vibrate_alert()` on receiving a `sedentary_alert` event — three short pulses of 0.2 s on / 0.2 s off. You can adjust the duration and number of pulses in [code.py](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3/code.py):

```python
def vibrate_alert():
    for _ in range(3):          # change to set number of pulses
        vibration_motor.value = True
        time.sleep(0.2)         # pulse duration in seconds
        vibration_motor.value = False
        time.sleep(0.2)         # gap between pulses
```

## Step 3 — Add the Data Foundry Script

The [Data Foundry Script](https://github.com/datacentricdesign/tutorials/tree/main/resources/T7.3/sedentary-detector.js) monitors the incoming activity scores and triggers the actuator after a configurable period of stillness.

1. In your Data Foundry project, go to the **Scripts** tab and create a new Script.
2. Paste the contents of `sedentary-detector.js` into the editor.
3. At the top of the script, fill in your channel names:

```javascript
const INCOMING_CHANNEL  = 'study/esense-data';   // sensor channel from Step 1
const ACTUATOR_CHANNEL  = 'your-esp32-channel';   // must match settings.py
```

4. Adjust the thresholds if needed:

```javascript
const STILL_THRESHOLD   = 0.02;   // activity_score below which a window counts as "still"
const STILL_TO_TRIGGER  = 150;    // consecutive still windows before triggering (150 × 2 s = 5 min)
const ACTIVE_TO_REARM   = 5;      // consecutive active windows needed to re-arm
```

5. Save and **enable** the script.

The script maintains a simple state machine: it counts consecutive still windows toward `STILL_TO_TRIGGER`, sends a `sedentary_alert` event to `ACTUATOR_CHANNEL` when the threshold is reached, then waits for `ACTIVE_TO_REARM` active windows before resetting. This prevents re-triggering immediately on the first slight movement after a long still period.

{: .note }
> **Remote iteration.** The threshold values live in the Data Foundry Script, not on the participant's device or the firmware. You can change `STILL_THRESHOLD` or `STILL_TO_TRIGGER`, save the script, and the next incoming data point runs against the new logic — no home visit, no reflashing, no reinstall. This is the practical payoff of keeping the classifier server-side during prototyping. As a concrete exercise: after your first deployment session, look at the `activity_score` values in the data table and ask whether the threshold you chose actually reflects stillness for this participant. Then update it remotely and observe whether the trigger pattern changes.

## Step 4 — Test the full pipeline

With the browser page streaming, the Data Foundry Script enabled, and the kit running, verify each link in the chain:

1. **Sensor → Data Foundry.** Sit still for 30 seconds. Confirm `activity_score` values are consistently below `STILL_THRESHOLD` in the Data Foundry data table.
2. **Script → Actuator.** Sit still for longer than `STILL_TO_TRIGGER × 2` seconds (default: 5 minutes). The kit should vibrate. Check the script log in Data Foundry for the "Trigger sent" message.
3. **Re-arm.** Move your head actively for a few seconds. The script should log "Detector re-armed". Sitting still again should eventually trigger another vibration.

If the actuator does not respond, check:
- The serial monitor on the kit — is it printing received messages?
- The OOCSI channel names match exactly in all three places (browser page, script, `settings.py`).
- The Data Foundry Script is enabled and not in an error state.

## Step 5 — Deploy and monitor

Give the participant the kit and the URL of the hosted page you published in Step 1 — they open it in Chrome, connect the earbud, and start streaming. Change the **Activity label** in the OOCSI panel from `TEST` to `DEPLOYED` before handing over, so study-phase data is tagged correctly in the dataset from the first session.

After 24 hours, open the Data Foundry data table and look for:

- **Gaps in the time-series.** A missing row every 2 seconds most likely means the browser tab was backgrounded or the laptop went to sleep — Web Bluetooth suspends when the tab is not active. Note when gaps occurred and how long they lasted.
- **Low `samples` counts.** Each window should contain around 100 packets (50 Hz × 2 s). A window with significantly fewer suggests BLE packet loss. Compare these against the gap timestamps to see whether they cluster.
- **Unexpected `activity_score` spikes.** A single high value in an otherwise still session may indicate the participant briefly moved, or it may be a BLE glitch inflating the variance calculation. Context from the `gyro` fields can help distinguish the two.

Once you have identified one data quality issue, make a corresponding change to the script — for example, tightening the minimum `samples` guard, or adjusting the threshold — and redeploy without visiting the participant.

## Reflect

Before drawing conclusions from your deployment data, consider:

1. **Proxy validity.** Head stillness is a proxy for sedentary behaviour. Under what conditions does this proxy break down? (Working at a desk with a fixed screen, sleeping, wearing the earbud while watching TV.)

2. **The browser as a data collection platform.** Web Bluetooth gave you a no-install, cross-device sensor interface. What did you lose compared to a native app or a dedicated probe? Did the tab-suspension issue affect your data?

3. **Windowing as a design decision.** The 2-second window and the 5-minute stillness threshold were chosen for this tutorial. What would change if you used a 10-second window? A 15-minute threshold? How would you choose these values for a real study?

4. **Participant response.** Did the vibration change behaviour, or did participants quickly learn to ignore it? What would you change about the feedback to make it more effective — its timing, intensity, or modality?

5. **Remote iteration in practice.** You changed a threshold without visiting the participant. What are the risks of doing this during a live study? What would you want to log to be confident a remote change had the intended effect?
